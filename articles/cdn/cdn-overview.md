## <a name="how-it-works"></a><span data-ttu-id="dcbe6-113">Принцип работы</span><span class="sxs-lookup"><span data-stu-id="dcbe6-113">How it works</span></span>
![Обзор сети доставки содержимого](./media/cdn-overview/cdn-overview.png)

1. <span data-ttu-id="dcbe6-115">Пользователь (Alice) запрашивает файл (ресурс), обращаясь к нему по URL-адресу со специальным доменным именем, например `<endpointname>.azureedge.net`.</span><span class="sxs-lookup"><span data-stu-id="dcbe6-115">A user (Alice) requests a file (also called an asset) using a URL with a special domain name, such as `<endpointname>.azureedge.net`.</span></span>  <span data-ttu-id="dcbe6-116">Служба DNS перенаправляет запрос на оптимальную точку присутствия (POP).</span><span class="sxs-lookup"><span data-stu-id="dcbe6-116">DNS routes the request to the best performing Point-of-Presence (POP) location.</span></span>  <span data-ttu-id="dcbe6-117">Обычно используется точка присутствия, которая географически ближе к пользователю.</span><span class="sxs-lookup"><span data-stu-id="dcbe6-117">Usually this is the POP that is geographically closest to the user.</span></span>
2. <span data-ttu-id="dcbe6-118">Если в кэше пограничных серверов этой точки присутствия нет нужного файла, пограничный сервер запрашивает файл у источника содержимого.</span><span class="sxs-lookup"><span data-stu-id="dcbe6-118">If the edge servers in the POP do not have the file in their cache, the edge server requests the file from the origin.</span></span>  <span data-ttu-id="dcbe6-119">Источником может служить веб-приложение Azure, облачная служба Azure, учетная запись хранилища Azure или любой общедоступный веб-сервер.</span><span class="sxs-lookup"><span data-stu-id="dcbe6-119">The origin can be an Azure Web App, Azure Cloud Service, Azure Storage account, or any publicly accessible web server.</span></span>
3. <span data-ttu-id="dcbe6-120">Источник содержимого возвращает пограничному серверу нужный файл и необязательные заголовки HTTP с информацией о сроке жизни (TTL) этого файла.</span><span class="sxs-lookup"><span data-stu-id="dcbe6-120">The origin returns the file to the edge server, including optional HTTP headers describing the file's Time-to-Live (TTL).</span></span>
4. <span data-ttu-id="dcbe6-121">Пограничный сервер помещает файл в кэш, и одновременно передает его пользователю (Alice).</span><span class="sxs-lookup"><span data-stu-id="dcbe6-121">The edge server caches the file and returns the file to the original requestor (Alice).</span></span>  <span data-ttu-id="dcbe6-122">Файл будет храниться в кэше на пограничном сервере до истечения срока жизни.</span><span class="sxs-lookup"><span data-stu-id="dcbe6-122">The file remains cached on the edge server until the TTL expires.</span></span>  <span data-ttu-id="dcbe6-123">Если источник не указывает срок жизни, по умолчанию принимается 7 дней.</span><span class="sxs-lookup"><span data-stu-id="dcbe6-123">If the origin didn't specify a TTL, the default TTL is seven days.</span></span>
5. <span data-ttu-id="dcbe6-124">После этого другой пользователь обращается к тому же файлу, используя тот же URL-адрес, и получает перенаправление на ту же точку присутствия.</span><span class="sxs-lookup"><span data-stu-id="dcbe6-124">Additional users may then request the same file using that same URL, and may also be directed to that same POP.</span></span>
6. <span data-ttu-id="dcbe6-125">Если срок жизни файла еще не истек, пограничный сервер возвращает файл из кэша.</span><span class="sxs-lookup"><span data-stu-id="dcbe6-125">If the TTL for the file hasn't expired, the edge server returns the file from the cache.</span></span>  <span data-ttu-id="dcbe6-126">Такая схема повышает скорость взаимодействия с пользователем и устраняет задержки.</span><span class="sxs-lookup"><span data-stu-id="dcbe6-126">This results in a faster, more responsive user experience.</span></span>
