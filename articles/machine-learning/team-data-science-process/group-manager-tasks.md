# <a name="group-manager-tasks"></a>Задачи менеджера группы

В этой статье описаны задачи, которые менеджер группы должен выполнить для своего подразделения обработки и анализа данных. Целью является создание среды для совместной работы группы, которая стандартизирует [командный процесс обработки и анализа данных](overview.md). Роли и задачи сотрудников в области стандартизации обработки и анализа данных описаны в статье [Team Data Science Process roles and tasks](roles-tasks.md) (Роли и задачи в командном процессе обработки и анализа данных).

**Менеджер группы** является руководителем всего подразделения обработки и анализа данных на предприятии. В подразделение обработки и анализа данных может входить несколько команд, каждая из которых выполняет несколько проектов по обработке и анализу данных для разных подразделений компании. Менеджер группы может делегировать свои задачи заместителям, но все задачи для этой роли остаются неизменными. Существует шесть основных задач, которые представлены на следующей схеме.

![0](./media/group-manager-tasks/tdsp-group-manager.png)


>[AZURE.NOTE] В инструкциях ниже описаны шаги по организации рабочей среды для процессов обработки и анализа данных (TDSP) с применением VSTS. Мы объясняем, как выполнить эти задачи с помощью VSTS, так как на базе именно этой платформы мы реализуем TDSP в Майкрософт. Если ваша группа использует для размещения кода другую платформу, задачи менеджера группы все равно не меняются. Но способ выполнения этих задач будет отличаться.

1. Настройка **сервера Visual Studio Team Services (VSTS)** для группы.
2. Создание **командного проекта группы** на сервере Visual Studio Team Services (для пользователей VSTS)
3. Создание репозитория **GroupProjectTemplate**
4. Создание репозитория **GroupUtilities**
5. Заполнение репозиториев **GroupProjectTemplate** и **GroupUtilities** на сервере VSTS данными из репозиториев TDSP.
6. Настройка **средств контроля безопасности** для доступа членов команды к репозиториям GroupProjectTemplate и GroupUtilities.

Ниже подробно описан каждый из этих шагов. Но сначала мы хотим напомнить вам некоторые условные обозначения и предварительные условия для работы с репозиториями.

### <a name="abbreviations-for-repositories-and-directories"></a>Аббревиатуры, используемые в репозиториях и каталогах

В этом руководстве используются сокращенные имена для репозиториев и каталогов. Эти условные обозначения упрощают описание и понимание операций с репозиториями и каталогами. В следующих разделах используются эти обозначения.

- **G1** — репозиторий шаблонов проектов, подготовленный и управляемый командой TDSP корпорации Майкрософт.
- **G2** — репозиторий служебных программ, подготовленный и управляемый командой TDSP корпорации Майкрософт.
- **R1** — репозиторий GroupProjectTemplate в Git, который вы настраиваете на сервере VSTS для вашей группы.
- **R2** — репозиторий GroupUtilities в Git, который вы настраиваете на сервере VSTS для вашей группы.
- **LG1** и **LG2** — локальные каталоги на компьютере, в которые клонируются репозитории G1 и G2, соответственно.
- **LR1** и **LR2** — локальные каталоги на компьютере, в которые клонируются репозитории R1 и R2, соответственно.

### <a name="pre-requisites-for-cloning-repositories-and-checking-code-in-and-out"></a>Необходимые условия для клонирования репозиториев и проверки кода на входе и на выходе
 
- На вашем компьютере должна быть установлена система Git. Если вы используете виртуальную машину обработки и анализа данных, система Git уже установлена и вы можете приступать к работе. Если система не установлена, см. [описание платформ и средств](platforms-and-tools.md#appendix).  
- Если вы используете **виртуальную машину для обработки и анализа данных Windows**, установите на компьютере [диспетчер учетных данных Git (GCM)](https://github.com/Microsoft/Git-Credential-Manager-for-Windows). Откройте файл README.md, прокрутите его до раздела **Download and Install** (Скачивание и установка) и выберите *latest installer* (последний установщик). Это действие открывает страницу самого свежего установщика. Загрузите EXE-файл установщика и запустите его. 
- Если вы используете **виртуальную машину для обработки и анализа данных Linux**, создайте открытый ключ SSH для этой виртуальной машины и добавьте его на сервер VSTS группы. Дополнительные сведения об SSH см. в разделе о **создании открытого ключа SSH** в [описании платформ и средств](platforms-and-tools.md#appendix). 


## <a name="1-create-account-on-vsts-server"></a>1. Создание учетной записи на сервере VSTS

На сервере VSTS размещаются следующие репозитории.

- **Общие репозитории группы**. Это репозитории общего назначения, которые могут использоваться несколькими командами в пределах группы для разных проектов обработки и анализа данных. Например, к этой категории относятся репозитории *GroupProjectTemplate* и *GroupUtilities*.
- **Репозитории команд**. Это репозитории для отдельных команд группы. Они выделяются под конкретные потребности команды и могут использоваться в нескольких проектах, выполняемых этой командой, но не достаточно универсальны для того, чтобы быть полезными для других команд группы обработки и анализа данных. 
- **Репозитории проектов**. Это репозитории, созданные для конкретных проектов. Обычно они недостаточно универсальны, чтобы использоваться для нескольких проектов или несколькими командами группы обработки и анализа данных.


### <a name="setting-up-the-vsts-server-sign-into-your-microsoft-account"></a>Настройка сервера VSTS. Вход в учетную запись Майкрософт
    
Откройте [Visual Studio Online](https://www.visualstudio.com/), нажмите кнопку **Вход** в правом верхнем углу и войдите в учетную запись Майкрософт. 
    
![1](./media/group-manager-tasks/login.PNG)

Если у вас нет учетной записи Майкрософт, нажмите кнопку **Зарегистрируйтесь сейчас**, создайте учетную запись Майкрософт и выполните вход с ее помощью. 

Если у вашей организации есть подписка MSDN или Visual Studio, нажмите зеленый блок **Войти в рабочую или учебную учетную запись** и выполните вход с использованием учетных данных этой подписки. 
        
![2](./media/group-manager-tasks/signin.PNG)


        
После входа нажмите кнопку **Создать учетную запись** в правом верхнем углу, как показано на следующем рисунке.
        
![3](./media/group-manager-tasks/create-account-1.PNG)
        
Введите для создаваемого VSTS сервера следующие сведения в мастере **создания учетной записи**. 

- **URL-адрес сервера**. Вместо строки *mysamplegroup* укажите *имя сервера*. URL-адрес сервера будет сформирован так: *https://\<имя_сервера\>.visualstudio.com*. 
- **Управление кодом с помощью:** выберите значение **_Git_**.
- **Имя проекта:** введите имя *GroupCommon*. 
- **Организуйте работу с помощью следующих средств:** выберите значение *Agile*.
- **Проекты будут размещены в:** выберите нужное географическое расположение. В нашем примере это *юго-центральный регион США*. 
        
![4](./media/group-manager-tasks/fill-in-account-information.png)

>[AZURE.NOTE] Если после нажатия кнопки **Создать учетную запись** вы увидите следующее всплывающее окно, выберите действие **Изменить сведения**, чтобы отобразить все поля по порядку.

![5](./media/group-manager-tasks/create-account-2.png)


Нажмите кнопку **Продолжить**.

## <a name="4-create-the-groupprojecttemplate-r1-repository"></a>4. Создание репозитория GroupProjectTemplate (R1)

Настройка репозиториев для группового сервера VSTS состоит из двух задач.

- Переименование создаваемого по умолчанию репозитория **GroupCommon** в ***GroupProjectTemplate***.
- Создание репозитория **GroupUtilities** на сервере VSTS внутри командного проекта **GroupCommon**. 

Инструкции для выполнения первой задачи размещены в этом разделе после информации о соглашениях об именованиях, наших репозиториях и каталогах. Инструкции для второй задачи вы найдете в шаге 4 следующего раздела.

### <a name="rename-the-default-groupcommon-repository"></a>Переименование стандартного репозитория GroupCommon

Чтобы переименовать созданный по умолчанию репозиторий **GroupCommon** в *GroupProjectTemplate* (обозначенный в этом руководстве как **R1**), выполните следующие действия.
    
- Нажмите кнопку **Совместная работа над кодом** на странице командного проекта **GroupCommon**. При этом откроется страница репозитория Git по умолчанию для командного проекта **GroupCommon**. Этот репозиторий Git пока пуст. 

![10](./media/group-manager-tasks/rename-groupcommon-repo-3.png)
        
- Щелкните репозиторий **GroupCommon** (выделен красным прямоугольником на следующем рисунке) в левом верхнем углу на странице репозитория Git **GroupCommon** и выберите элемент **Управление репозиториями**(выделен зеленой рамкой на следующем рисунке). Откроется **Панель управления**. 
- Выберите вкладку **Управление версиями** для командного проекта. 

![11](./media/group-manager-tasks/rename-groupcommon-repo-4.png)

- Щелкните многоточие (**...**) справа от репозитория **GroupCommon** на левой панели, а затем выберите **Переименовать репозиторий**. 

![12](./media/group-manager-tasks/rename-groupcommon-repo-5.png)
        
- Откроется **мастер переименования** для репозитория GroupCommon. Введите *GroupProjectTemplate* в поле **Имя репозитория** и щелкните **Переименовать**. 

![13.](./media/group-manager-tasks/rename-groupcommon-repo-6.png)



## <a name="5-seed-the-r1--r2-repositories-on-the-vsts-server"></a>5. Заполнение данными репозиториев R1 и R2 на сервере VSTS

На этом этапе процедуры вы внесете начальные данные в репозитории *GroupProjectTemplate* (R1) и *GroupUtilities* (R2), которые вы настроили в предыдущем разделе. Для заполнения используются данные из репозиториев ***ProjectTemplate*** (**G1**) и ***Utilities*** (**G2**), которые корпорация Майкрософт создала для поддержки командного процесса обработки и анализа данных. Операция заполнения завершится следующим состоянием:

- репозиторий R1 будет содержать тот же набор каталогов и шаблонов документов, что и G1;
- репозиторий R2 будет содержать набор служебных программ для обработки и анализа данных, разработанных корпорацией Майкрософт.

Процедура заполнения использует каталоги на локальной виртуальной машине обработки и анализа данных в качестве промежуточных расположений для хранения и обработки. Вот список шагов, которые мы обсудим далее в этом разделе:

- клонирование G1 и G2 в -> LG1 и LG2;
- клонирование R1 и R2 в -> LR1 и LR2;
- копирование файлов из LG1 и LG2 в -> LR1 и LR2;
- (необязательно) настройка каталогов LR1 и LR2;
- добавление содержимого из LR1 и LR2 в -> R1 & R2.


### <a name="clone-g1--g2-repositories-to-your-local-dsvm"></a>Клонирование репозиториев G1 и G2 в локальную систему виртуальной машины для обработки и анализа данных

На этом шаге вы клонируете репозитории ProjectTemplate (G1) и Utitlities (G2), предназначенные для поддержки командного процесса обработки и анализа данных, в папки LG1 и LG2 на локальной виртуальной машине для обработки и анализа данных.

- Создайте каталог, который будет корневым для всех клонированных репозиториев. 
    -  На виртуальной машине для обработки и анализа данных Windows создайте каталог *C:\GitRepos\TDSPCommon*. 
    -  На виртуальной машине для обработки и анализа данных Linux создайте каталог *GitRepos\TDSPCommon* в домашнем каталоге. 

- Запустите следующий набор команд из каталога *GitRepos\TDSPCommon*.

    `git clone https://github.com/Azure/Azure-TDSP-ProjectTemplate`<br>
    `git clone https://github.com/Azure/Azure-TDSP-Utilities`
        
![14](./media/group-manager-tasks/two-folder-cloned-from-TDSP-windows.PNG)

- В наших условных обозначениях эти операции выглядят так: 
    - клонирование G1 в -> LG1;
    - клонирование G2 в -> LG2.
- Когда клонирование завершится, вы найдете два новых каталога (_ProjectTemplate_ и _Utilities_) в каталоге **GitRepos\TDSPCommon**. 

### <a name="clone-r1--r2-repositories-to-your-local-dsvm"></a>Клонирование репозиториев R1 и R2 в локальную систему виртуальной машины для обработки и анализа данных

На этом шаге вы клонируете репозитории GroupProjectTemplate (R1) и GroupUtilities (R2) в локальные каталоги (которые мы обозначаем LR1 и LR2, соответственно) в каталоге **GitRepos\GroupCommon** на локальной виртуальной машины для обработки и анализа данных.

- Чтобы получить URL-адреса репозиториев R1 и R2, перейдите на домашнюю страницу **GroupCommon** в системе VSTS. Обычно эти URL-адреса выглядят так: *https://\<имя_сервера_VSTS\>.visualstudio.com/GroupCommon*. 
- Щелкните элемент **Код**. 
- Выберите репозитории **GroupProjectTemplate** и **GroupUtilities**. Скопируйте и сохраните оба URL-адреса (с протоколом HTTPS для Windows или SSH для Linux) из элемента **URL-адрес клона** и примените их в следующих скриптах.  

![15](./media/group-manager-tasks/find_https_ssh_2.PNG)

- Перейдите в каталог **GitRepos\GroupCommon** в локальной системе виртуальной машины для обработки и анализа данных Windows или Linux и выполните в ней соответствующий набор команд для клонирования R1 и R2 в этот каталог.
        
Вот скрипты этой операции для Windows и Linux:

    # Windows DSVM

    git clone <the HTTPS URL of the GroupProjectTemplate repository>
    git clone <the HTTPS URL of the GroupUtilities repository>

![16](./media/group-manager-tasks/clone-two-empty-group-reo-windows-2.PNG)

    # Linux DSVM

    git clone <the SSH URL of the GroupProjectTemplate repository>
    git clone <the SSH URL of the GroupUtilities repository>

![17](./media/group-manager-tasks/clone-two-empty-group-reo-linux-2.PNG)

>[AZURE.NOTE] Вы должны получить предупреждающие сообщения о том, что LR1 и LR2 пусты.    

- В наших условных обозначениях эти операции выглядят так: 
    - клонирование R1 в -> LR1;
    - клонирование R2 в -> LR2.   


### <a name="seed-your-groupprojecttemplate-lr1-and-grouputilities-lr2"></a>Заполнение данными каталогов GroupProjectTemplate (LR1) и GroupUtilities (LR2)

На локальном компьютере скопируйте содержимое каталогов ProjectTemplate и Utilities (за исключением метаданных в каталогах .git), расположенных в каталоге GitRepos\TDSPCommon, в каталоги GroupProjectTemplate и GroupUtilities, расположенные в каталоге **GitRepos\ GroupCommon**. Ниже приведены две задачи, которые нужно выполнить на этом шаге.

- Копирование файлов из GitRepos\TDSPCommon\ProjectTemplate (**LG1**) в GitRepos\GroupCommon\GroupProjectTemplate (**LR1**) 
- Копирование файлов из GitRepos\TDSPCommon\Utilities (**LG2** в GitRepos\GroupCommon\Utilities (**LR2**). 

Чтобы выполнить эти две задачи, запустите следующие скрипты в консоли PowerShell (Windows) или в консоли оболочки (Linux). Вы получите запрос на ввод полного пути к каталогам LG1, LR1, LG2 и LR2. Введенные данные проверяются. Если вы введете несуществующий путь к каталогу, запрос на ввод появится снова. 

    # Windows DSVM      
    
    wget "https://raw.githubusercontent.com/Azure/Azure-MachineLearning-DataScience/master/Misc/TDSP/tdsp_local_copy_win.ps1" -outfile "tdsp_local_copy_win.ps1"
    .\tdsp_local_copy_win.ps1 1

![18](./media/group-manager-tasks/copy-two-folder-to-group-folder-windows-2.PNG)

Вы можете убедиться, что в результате этих действий файлы из каталогов LG1 и LG1 (за исключением файлов в каталоге .git) скопированы в каталоги LR1 и LR2, соответственно.

![19](./media/group-manager-tasks/copy-two-folder-to-group-folder-windows.PNG)

    # Linux DSVM

    wget "https://raw.githubusercontent.com/Azure/Azure-MachineLearning-DataScience/master/Misc/TDSP/tdsp_local_copy_linux.sh"
    bash tdsp_local_copy_linux.sh 1

![20](./media/group-manager-tasks/copy-two-folder-to-group-folder-linux-2.PNG)
        
Теперь все файлы из двух каталогов (за исключением файлов в каталоге .git) скопированы в каталоги GroupProjectTemplate и GroupUtilities, соответственно.
    
![21](./media/group-manager-tasks/copy-two-folder-to-group-folder-linux.PNG)

- В наших условных обозначениях эти операции выглядят так:
    - копирование файлов из LG1 в -> LR1;
    - копирование файлов из LG2 в -> LR2.

### <a name="option-to-customize-the-contents-of-lr1--lr2"></a>Дополнительная настройка содержимого LR1 и LR2
    
Если вы хотите настроить содержимое LR1 и LR2 для конкретных целей вашей группы, это лучше всего сделать именно на этом этапе процедуры. Вы можете редактировать шаблоны документов, изменить структуру каталогов или добавить новые программы, разработанные вашей группой или полезные для ее работы. 

### <a name="add-the-contents-in-lr1--lr2-to-r1--r2-on-group-server"></a>Добавление содержимого из LR1 и LR2 в репозитории R1 и R2 на сервере группы

Теперь вам осталось перенести содержимое каталогов LR1 и LR2 репозитории R1 и R2. Для этого запустите приведенные ниже команды git bash в Windows PowerShell или Linux. 

Следующие команды выполняются из каталога GitRepos\GroupCommon\GroupProjectTemplate.

    git status
    git add .
    git commit -m"push from DSVM"
    git push

Параметр -m позволяет указать сообщение для фиксации в git.

![22](./media/group-manager-tasks/push-to-group-server-2.PNG)

Вы увидите, что все файлы мгновенно синхронизируются в репозитории GroupProjectTemplate на сервере VSTS для вашей группы.

![23](./media/group-manager-tasks/push-to-group-server-showed-up-2.PNG)

Наконец, перейдите в каталог **GitRepos\GroupCommon\GroupUtilities** и еще раз выполните ту же команду git bash.

    git status
    git add .
    git commit -m"push from DSVM"
    git push

>[AZURE.NOTE] Если вы впервые фиксируете изменения в репозиторий Git, необходимо настроить глобальные параметры *user.name* и *user.email* перед выполнением команды `git commit`. Выполните эти две команды:
        
    git config --global user.name <your name>
    git config --global user.email <your email address>
 
>Если вы фиксируете изменения в нескольких репозиториях Git, используйте одинаковые имя и адрес электронной почты во время фиксации в каждом из них. Использование одних и тех же имени и адреса электронной почты пригодится вам позднее, когда вы создадите панель мониторинга PowerBI для отслеживания своих действий в нескольких репозиториях Git.


- В наших условных обозначениях эти операции выглядят так:
    - добавление содержимого из LR1 в -> R1;
    - добавление содержимого из LR2 в -> R2.

## <a name="6-add-group-members-to-the-group-server"></a>6. Добавление членов группы на сервер группы

На домашней странице сервера VSTS группы выберите **значок шестеренки** рядом с именем пользователя в правом верхнем углу. Затем выберите вкладку **Безопасность**. Здесь вы можете добавлять участников группы с разными разрешениями.
