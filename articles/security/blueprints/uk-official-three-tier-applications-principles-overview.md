# <a name="national-cyber-security-centre-cloud-security-principles-overview"></a>Обзор принципов облачной безопасности Национального центра кибербезопасности (NCSC)


> [!NOTE]
> Эти принципы безопасности определены Национальным центром кибербезопасности (Соединенное Королевство) (NCSC). Чтобы получить сведения о процедурах тестирования и рекомендации для каждого принципа безопасности, обратитесь к [документации по NCSC](https://www.ncsc.gov.uk/guidance/implementing-cloud-security-principles).



## <a name="ncsc-cloud-security-principle-1"></a>Принцип облачной безопасности NCSC 1
### <a name="data-in-transit-protection"></a>Защита данных в процессе передачи
Сети, передающие данные пользователей, должны быть надлежащим образом защищены от несанкционированного доступа и перехвата трафика.

Этого нужно достичь путем сочетания:

- защиты сети — исключение возможности перехвата данных злоумышленником и
- шифрования — исключение возможности чтения данных злоумышленником.


**Обязанности:** `Shared`


|||
|---|---|
| **Клиент** | Azure Blueprint настраивает для ресурсов возможность подключения только по защищенным протоколам. Компонент WAF шлюза приложений настроен для приема коммуникаторов из внешних приложений через HTTPS/TLS и связи с серверным пулом только через HTTPS/TLS. Службы удаленного рабочего стола настроены для использования безопасных подключений. Для защиты веб-трафика между AppGateway и Azure используется VPN. |
| **Поставщик&nbsp;(Microsoft&nbsp;Azure)** | Azure использует стандартный протокол TLS 1.2 с 2048-разрядными ключами шифрования RSA/SHA256, как рекомендовано CESG/NCSC, для шифрования сообщений между клиентом и облаком, а также между системами Azure и центрами обработки данных. Например, когда администраторы используют портал Microsoft Azure для управления службой в своей организации, данные, передаваемые между порталом и устройством администратора, отправляются по зашифрованному каналу TLS. Когда пользователь электронной почты подключается к Outlook.com с помощью стандартного веб-браузера, HTTPS-соединение обеспечивает безопасный канал для приема и отправки электронной почты.<br /> <br /> Azure предлагает своим клиентам ряд механизмов для защиты своих данных и трафика. Функции управления сертификатами, встроенные в Azure, предоставляют администраторам гибкость при настройке сертификатов и ключей шифрования для служб управления, отдельных служб, сеансов SSH, VPN-подключений, подключений удаленного рабочего стола (RDP) и других функций. <br /><br /> Разработчики могут использовать поставщики криптографических услуг (CSP), встроенные в Microsoft .NET Framework, для доступа к алгоритмам AES, а также функциональные возможности SHA-2 для решения таких задач, как проверка цифровых подписей. Azure Key Vault помогает клиентам защищать криптографические ключи и секреты, сохраняя их в аппаратных модулях безопасности (HSM). |


 ## <a name="ncsc-cloud-security-principle-2"></a>Принцип облачной безопасности NCSC 2
### <a name="asset-protection-and-resilience"></a>Защита ресурсов и отказоустойчивость
Данные пользователей, а также ресурсы, которые хранят их или обрабатывают, должны быть защищены от физического вмешательства, потери, повреждения или перехвата.

Ниже приведены аспекты, которые следует учитывать.

1. Физическое расположение и правовая юрисдикция.
2. Безопасность центра обработки данных.
3. Защита неактивных данных.
4. Очистка данных.
5. Утилизация оборудования.
6. Физическая отказоустойчивость и доступность


 ## <a name="ncsc-cloud-security-principle-21"></a>Принцип облачной безопасности NCSC 2.1
### <a name="physical-location-and-legal-jurisdiction"></a>Физическое расположение и правовая юрисдикция
Чтобы понять юридические обстоятельства, при которых ваши данные могут быть доступны без вашего согласия, вы должны определить расположения, в которых они хранятся, обрабатываются и управляются.
Вам также нужно будет понять, как применяются средства управления обработкой данных в рамках службы (в отношении законодательства Соединенного Королевства). Неправильная защита пользовательских данных может привести к правовым и нормативным санкциям или к подрыву репутации.


**Обязанности:** `Customer`

> [!NOTE]
> Службы Azure развертываются на региональном уровне, и клиенты могут настраивать определенные службы Azure для хранения данных клиентов только в одном регионе. Microsoft Azure предоставляет список доступных центров обработки данных для обеспечения доступности и надежности в глобальном масштабе. Все центры обработки данных Azure сертифицированы по стандарту ISO/IEC 27001:2013. Географический регион Соединенного Королевства состоит из 2 регионов: южная часть Соединенного Королевства и западная часть Соединенного Королевства.

|||
|---|---|
| **Клиент** | Azure Blueprint запрашивает администратора, в каком регионе следует развернуть ресурсы Azure. Рекомендуемыми регионами для развертывания являются южная часть Соединенного Королевства или западная часть Соединенного Королевства. |
| **Поставщик&nbsp;(Microsoft&nbsp;Azure)** | Не применяется |


 ## <a name="ncsc-cloud-security-principle-22"></a>Принцип облачной безопасности NCSC 2.2
### <a name="datacentre-security"></a>Безопасность центра обработки данных
Расположения, используемые для предоставления облачных служб, нуждаются в физической защите от несанкционированного доступа, незаконного изменения, кражи или перенастройки систем. Недостаточная защита может привести к раскрытию, изменению или потере данных.


**Обязанности:** `Microsoft Azure`


|||
|---|---|
| **Клиент** | Клиенты не имеют физического доступа к каким-либо системным ресурсам в центрах обработки данных Azure. Меры по защите данных в центрах обработки данных реализует и контролирует Microsoft Azure. Этот принцип наследуется от Microsoft Azure. |
| **Поставщик&nbsp;(Microsoft&nbsp;Azure)** | Microsoft Azure реализует этот принцип от имени клиентов. Microsoft Azure работает в географически распределенных объектах Майкрософт, совместно используя пространство и служебные программы с другими веб-службами Майкрософт. Каждый объект предназначен для круглосуточной работы и использует различные отраслевые меры для защиты от сбоев питания, физического вмешательства и сбоев сети. Эти центры обработки данных соответствуют отраслевым стандартам (например, ISO 27001) для обеспечения физической безопасности и доступности. Контроль, управление и администрирование осуществляется операционным персоналом Майкрософт. <br /> <br /> Клиенты Azure могут быть уверены в том, что средства физической безопасности применяются во всех центрах обработки данных Azure благодаря сертификатам стандарта ISO/IEC 27001:2013 во всех центрах обработки данных Azure. Географический регион Соединенного Королевства состоит из 2 регионов: южная часть Соединенного Королевства и западная часть Соединенного Королевства. |


 ## <a name="ncsc-cloud-security-principle-23"></a>Принцип облачной безопасности NCSC 2.3
### <a name="data-at-rest-protection"></a>Защита неактивных данных
Чтобы данные не были доступны для несанкционированных лиц с физическим доступом к инфраструктуре, пользовательские данные, хранящиеся в службе, должны быть защищены независимо от носителя, на котором они хранятся. Без соответствующих мер данные могут быть непреднамеренно раскрыты на отклоненных, потерянных или украденных носителях.


**Обязанности:** `Shared`

> [!NOTE]
> Все решения для шифрования, которые Microsoft Azure предоставляет своим клиентам, легко интегрируются с хранилищем Azure Key Vault, что позволяет легко управлять ключами шифрования.

|||
|---|---|
| **Клиент** | Конфиденциальность и целостность всех хранилищ BLOB-объектов, развернутых этим решением Azure Blueprint, защищены с помощью шифрования службы хранилища Azure (SSE). SSE защищает неактивные данные в учетных записях хранения Azure с использованием 256-битного шифрования AES. |
| **Поставщик&nbsp;(Microsoft&nbsp;Azure)** | Azure предлагает широкий спектр возможностей шифрования, предоставляя клиентам достаточную гибкость при выборе решений, наилучшим образом удовлетворяющих их потребности. Azure Key Vault позволяет удобно и рентабельно контролировать ключи, используемые облачными приложениями и службами для шифрования данных. Шифрование дисков Azure позволяет клиентам шифровать виртуальные машины. Шифрование службы хранилища Azure позволяет шифровать все данные, помещенные в клиентскую учетную запись хранения. |


 ## <a name="ncsc-cloud-security-principle-24"></a>Принцип облачной безопасности NCSC 2.4
### <a name="data-sanitisation"></a>Очистка данных
Процесс подготовки ресурсов, переноса и отмены подготовки ресурсов не должен приводить к несанкционированному доступу к пользовательским данным.

Недостаточная очистка данных может привести к следующим последствиям:

- Данные пользователя будут храниться поставщиком услуг неопределенное время.
- Данные пользователя будут доступны другим пользователям службы, поскольку ресурсы повторно используются.
- Данные пользователя могут быть утеряны или скомпрометированы на отозванных, потерянных или украденных носителях.


**Обязанности:** `Microsoft Azure`


|||
|---|---|
| **Клиент** | Microsoft Azure предоставляет контрактные гарантии относительно полного удаления данных в Azure. Таким образом, этот принцип наследуется от Microsoft Azure. |
| **Поставщик&nbsp;(Microsoft&nbsp;Azure)** | Microsoft Azure следует процессу удаления NIST SP800-88r1 и классификации данных, соответствующей FIPS-199 Moderate. NIST обеспечивает подход безопасного удаления (с помощью встроенного ПО жесткого диска) для дисков, которые его поддерживают. Что касается нестираемых жестких дисков, Майкрософт уничтожает их и делает невозможным восстановление содержащейся на них информации (путем разборки, измельчения, размалывания или сжигания). Соответствующие средства утилизации зависят от типа ресурса. Записи об уничтожении сохраняются. Все службы Microsoft Azure используют утвержденные практики управления хранением и удалением носителей. <br />  <br /> По истечении срока действия или прекращении использования подписки на услугу клиент может связаться с компанией Майкрософт и сообщить, что следует сделать: <br /><br /> (1) Отключить учетную запись клиента, а затем удалить данные. <br /> (2) Либо хранить данные, размещенные в веб-службе, в учетной записи с ограниченными функциями не менее 90 дней после истечения срока действия или прекращения использования подписки клиента (период хранения), чтобы клиент мог извлечь данные. По истечении срока хранения Майкрософт отключит учетную запись клиента и удалит все данные. Кэшированные или резервные копии будут удалены в течение 30 дней после окончания периода хранения. |


 ## <a name="ncsc-cloud-security-principle-25"></a>Принцип облачной безопасности NCSC 2.5
### <a name="equipment-disposal"></a>Утилизация оборудования
После того как заканчивается срок службы оборудования, используемого для предоставления службы, оно должно быть утилизировано таким образом, чтобы это не компрометировало безопасность службы или данные пользователя, хранящиеся в службе.


**Обязанности:** `Microsoft Azure`


|||
|---|---|
| **Клиент** | Клиенты не имеют физического доступа к каким-либо системным ресурсам в центрах обработки данных Azure. Процедуры утилизации оборудования реализует и контролирует Microsoft Azure. Этот принцип наследуется от Microsoft Azure. |
| **Поставщик&nbsp;(Microsoft&nbsp;Azure)** | Microsoft Azure реализует этот принцип от имени клиентов. По истечении срока службы системы операционный персонал Майкрософт следует строгим процедурам обработки данных и процессам утилизации оборудования, чтобы гарантировать, что никакие аппаратные средства, которые могут содержать данные клиентов, не будут доступны недоверенным сторонам. Microsoft Azure следует процессу удаления NIST SP800-88r1 и классификации данных, соответствующей FIPS-199 Moderate. NIST обеспечивает подход безопасного удаления (с помощью встроенного ПО жесткого диска) для дисков, которые его поддерживают. Что касается нестираемых жестких дисков, Майкрософт уничтожает их и делает невозможным восстановление содержащейся на них информации (путем разборки, измельчения, размалывания или сжигания). Соответствующие средства утилизации зависят от типа ресурса. Записи об уничтожении сохраняются. Все службы Microsoft Azure используют утвержденные практики управления хранением и удалением носителей. |


 ## <a name="ncsc-cloud-security-principle-26"></a>Принцип облачной безопасности NCSC 2.6
### <a name="physical-resilience-and-availability"></a>Физическая отказоустойчивость и доступность
Службы имеют разный уровень отказоустойчивости, что будет влиять на их способность нормально работать в случае сбоев, инцидентов или атак. Служба без гарантии доступности может стать недоступной, возможно, в течение длительного времени, независимо от влияния на ваш бизнес.


**Обязанности:** `Shared`

> [!NOTE]
> Если клиент настроит Microsoft Azure соответствующим образом, включив Azure Site Recovery и альтернативное хранилище данных в другом географически расположенном центре обработки данных, Microsoft Azure может поддерживать непрерывную работу ресурсов, развернутых клиентами.

|||
|---|---|
| **Клиент** | Клиент несет ответственность за создание альтернативного места хранения. В заявлении клиента об осуществлении контроля должна быть указана способность клиента действовать в случае инцидента. |
| **Поставщик&nbsp;(Microsoft&nbsp;Azure)** | Microsoft Azure имеет одобренные Национальным центром кибербезопасности (NCSC) центры обработки данных в разных географических расположениях (южная и западная часть Соединенного Королевства), чтобы обеспечить устойчивость и доступность. Клиент будет обязан резервировать ресурсы в альтернативном регионе, используя службу Azure Site Recovery. После настройки Azure Site Recovery Azure выполнит простой переход служб клиента на альтернативный центр обработки данных. |


 ## <a name="ncsc-cloud-security-principle-3"></a>Принцип облачной безопасности NCSC 3
### <a name="separation-between-users"></a>Разделение между пользователями
Злоумышленник или скомпрометированный пользователь службы не должен влиять на службу или данные другого пользователя.

Факторы, влияющие на разделение пользователей, включают в себя:

- где реализованы механизмы разделения, на что в значительной степени влияет модель обслуживания (например, IaaS, PaaS, SaaS);
- с кем вы совместно используете службу, что продиктовано моделью развертывания (например, общедоступная, частная или облачная);
- уровень надежности, доступный при осуществлении контроля за разделением.

> [!NOTE]
> При использовании службы IaaS вам следует рассмотреть возможность разделения, предоставляемого компонентами вычислений, хранения и сети. Кроме того, службы SaaS и PaaS на основе IaaS могут наследовать некоторые из свойств разделения базовой инфраструктуры IaaS.

**Обязанности:** `Microsoft Azure`


|||
|---|---|
| **Клиент** | Microsoft Azure обеспечивает изоляцию каждого пользователя, чтобы вредоносный или скомпрометированный пользователь не влиял на службу или данные другого пользователя. Таким образом, этот принцип наследуется от Microsoft Azure. |
| **Поставщик&nbsp;(Microsoft&nbsp;Azure)** | По причине виртуальной природы облачных серверов пользователей парадигма физического разделения больше не применяется. Microsoft Azure была разработана, чтобы помочь идентифицировать риски, присущие мультитенантной среде, и противостоять им. Хранение и обработка данных логически разделены между пользователями Azure с использованием Active Directory и функций, специально разработанных для мультитенантных служб, чтобы пользовательские данные, хранящиеся в общедоступных центрах обработки данных Azure, не были доступны другой организации. <br /> <br /> Основой для любой общей облачной архитектуры является изоляция, предоставляемая каждому пользователю, чтобы злоумышленник или скомпрометированный пользователь не влиял на службу или данные другого пользователя. <br /> <br /> Дополнительные сведения о разделении клиентов компании Майкрософт см. в полном описании [таблицы обязанностей клиентов в рамках Azure Blueprint и принципов облачной безопасности NCSC](https://aka.ms/blueprintuk-gcrm). |


 ## <a name="ncsc-cloud-security-principle-4"></a>Принцип облачной безопасности NCSC 4
### <a name="governance-framework"></a>Структура системы управления
Поставщик услуг должен иметь структуру управления безопасностью, которая координирует управление службой и размещенной в ней информацией. Любые технические элементы управления, развернутые за пределами этой структуры, будут неэффективными.

 ## <a name="ncsc-cloud-security-principle-5"></a>Принцип облачной безопасности NCSC 5
### <a name="operational-security"></a>Операционная безопасность
Эксплуатация службы и управление ею должны осуществляться безопасно, чтобы обнаруживать, предотвращать атаки и препятствовать им. Хорошая эксплуатационная безопасность не должна требовать сложных, бюрократических, трудоемких или дорогостоящих процессов.

Следует учесть четыре элемента:

- Управление конфигурацией и изменениями. Следует убедиться, что изменения в системе прошли надлежащую проверку и авторизацию. Изменения не должны неожиданно менять свойства безопасности.
- Управление уязвимостями. Вы должны обнаруживать и устранять проблемы с безопасностью в составных компонентах.
- Защитный мониторинг. Вы должны принять меры для обнаружения атак и несанкционированных действий в службе.
- Управление инцидентами. Убедитесь, что вы можете реагировать на инциденты и восстанавливать безопасность и доступность службы.

Следует обнаруживать изменения, которые могут повлиять на безопасность службы, и управлять ими. Несанкционированные изменения должны обнаруживаться.
В тех случаях, когда в отношении изменения не осуществляется эффективное управление, в службе могут непреднамеренно возникнуть уязвимости безопасности. И даже там, где осуществляется отслеживание уязвимости, она может быть не полностью устранена.

Моделирование угроз — это командное упражнение, в котором участвуют менеджер операций, менеджеры программ и проектов, разработчики и тестировщики. Оно представляет собой ключевую задачу анализа безопасности, выполняемую для разработки решения. Члены команды используют средство моделирования угроз SDL для моделирования всех служб и проектов, как при их создании, так и при их обновлении для добавления новых функций и возможностей. Модели угроз охватывают весь код, подвергаемый атаке, в том числе написанный или лицензированный третьей стороной, с учетом всех границ доверия. Система STRIDE (спуфинг, незаконное изменение, отказ, раскрытие информации, отказ в обслуживании и повышение привилегий) используется для выявления и устранения угроз безопасности на ранней стадии процесса проектирования, прежде чем они могут повлиять на клиентов. |

В заявлении клиента о реализации средства управления должно быть указано исключение, чтобы отслеживать процессы, указанные в этой документации по цепочке поставок. |

 ## <a name="ncsc-cloud-security-principle-9"></a>Принцип облачной безопасности NCSC 9
### <a name="secure-user-management"></a>Безопасное управление пользователями
Ваш поставщик должен сделать доступными инструменты, чтобы вы могли безопасно управлять использованием службы. Интерфейсы и процедуры управления являются важной частью барьера безопасности, который предотвращает несанкционированный доступ и изменение ваших ресурсов, приложений и данных.

Ниже приведены аспекты, которые следует учитывать.

- Проверка подлинности пользователей в интерфейсах управления и каналах поддержки.
- Разделение и управление доступом в интерфейсах управления.



 ## <a name="ncsc-cloud-security-principle-91"></a>Принцип облачной безопасности NCSC 9.1
### <a name="authentication-of-users-to-management-interfaces-and-within-support-channels"></a>Проверка подлинности пользователей в интерфейсах управления и каналах поддержки
Чтобы поддерживать безопасность службы, пользователи должны пройти надлежащую проверку подлинности, прежде чем им разрешат выполнять действия по управлению, сообщать о неисправностях или запрашивать изменения в службе.
Эти действия могут выполняться через веб-портал управления службами или через другие каналы, такие как телефон или электронная почта. Они, вероятно, будут включать такие функции, как предоставление новых элементов обслуживания, управление учетными записями пользователей и пользовательскими данными.
Поставщики служб должны гарантировать, что все запросы управления, которые могут оказать влияние на безопасность, выполняются по защищенным и аутентифицированным каналам. Если пользователи не проходят надлежащую проверку подлинности, то самозванец может успешно выполнять привилегированные действия, ставя под угрозу безопасность службы или данных.


**Обязанности:** `Shared`


|||
|---|---|
| **Клиент** | Когда администраторы получают доступ к порталу Microsoft Azure для управления ресурсами Azure для своей организации, данные, передаваемые между порталом и устройством администратора, отправляются по зашифрованному каналу безопасности транспортного уровня (TLS) с использованием 2048-битных ключей шифрования RSA/SHA256, как рекомендовано CESG/НККК.  <br /> <br /> Azure Blueprint использует проверку подлинности Windows, удаленный рабочий стол и BitLocker. Эти компоненты можно настроить так, чтобы они полагались на проверенные криптографические модули FIPS 140. <br /> <br /> Azure Blueprint применяет разрешения логического доступа с помощью управления доступом на основе ролей, обеспечиваемого Azure Active Directory путем назначения пользователям ролей и Active Directory путем назначения пользователей группам безопасности и элементами управления на уровне операционной системы Windows. С помощью назначенных пользователям или группам ролей Azure Active Directory можно управлять логическим доступом к ресурсам в Azure на уровне ресурса, группы или подписки. Группы безопасности Active Directory управляют логическим доступом к ресурсам и функциям на уровне операционной системы. |
| **Поставщик&nbsp;(Microsoft&nbsp;Azure)** | Клиенты управляют своими ресурсами Azure через портал Azure, который обеспечивает доступ ко всем виртуальным машинам, базам данных, облачным службам и другим ресурсам, настроенным в учетной записи клиента. Веб-доступ к порталу Azure защищен с помощью подключений TLS 1.2 с использованием 2048-разрядных ключей шифрования RSA/SHA256, как рекомендовано CESG/NCSC. Предоставляются средства управления доступом на основе ролей, которые позволяют клиентам предоставлять ограниченный доступ к ресурсам управления Azure для конкретных пользователей и групп. |


 ## <a name="ncsc-cloud-security-principle-92"></a>Принцип облачной безопасности NCSC 9.2
### <a name="separation-and-access-control-within-management-interfaces"></a>Разделение и управление доступом в интерфейсах управления
Управление многими облачными службами осуществляется через веб-приложения или интерфейсы API. Эти интерфейсы являются основной частью обеспечения безопасности службы. Если пользователи не разделены должным образом в интерфейсах управления, один пользователь может повлиять на службу или изменить данные другого.
Вероятно, ваши привилегированные учетные записи имеют доступ к большим объемам данных. Предоставление доступа к таким данным только отдельным пользователям однозначно может помочь минимизировать ущерб, причиненный вредоносными пользователями, скомпрометированными учетными данными или скомпрометированными устройствами.
Управление доступом на основе ролей обеспечивает такой механизм и, вероятно, будет особенно важной возможностью для пользователей, управляющих большими развертываниями.
Предоставление интерфейсов управления в менее доступных сетях (например, сообществам, а не общедоступным сетям) затрудняет доступ злоумышленникам к ним, поскольку они сначала должны получить доступ к одной из этих сетей. Руководство по оценке рисков предоставления интерфейсов в различных типах сетей приведено в принципе 11.


**Обязанности:** `Shared`


|||
|---|---|
| **Клиент** | Azure Blueprint развертывает шлюз приложений, балансировщик нагрузки и настраивает правила группы безопасности сети для управления коммутациями на внешних границах и между внутренними подсетями. Функциональность пользователя отделена от функциональных возможностей системного управления с помощью обеспечения контроля логического доступа и системной архитектуры. Интерфейсы для функций управления системой отделены от пользовательских интерфейсов. Все возможности управления подключены через безопасный узел-бастион (jumpbox), расположенный в подсети управления, с правилами групп безопасности сети, чтобы ограничить доступ к рабочим ресурсам соответствующим образом. |
| **Поставщик&nbsp;(Microsoft&nbsp;Azure)** | Как описано выше, разделение пользователей изначально встроено в Azure. Azure Active Directory (Azure AD или AAD) может использоваться для предоставления каждому пользователю, который проходит аутентификацию на портале Azure, доступа только к тем ресурсам, которые они имеют право просматривать и контролировать. В результате различные учетные записи клиентов строго отделены друг от друга при управлении через общий портал Azure. |


 ## <a name="ncsc-cloud-security-principle-10"></a>Принцип облачной безопасности NCSC 10
### <a name="identity-and-authentication"></a>Идентификация и аутентификация
Весь доступ к интерфейсам служб должен предоставляться аутентифицированным и уполномоченным лицам.
Слабая аутентификация в этих интерфейсах может сделать возможным несанкционированный доступ к вашим системам, что приведет к краже или изменению данных, изменениям в вашей службе или отказу в обслуживании.
Важно отметить, что аутентификация должна происходить по защищенным каналам. Электронная почта, HTTP или телефон уязвимы к атакам перехвата и социальной инженерии.


**Обязанности:** `Shared`


|||
|---|---|
| **Клиент** | В Azure Blueprint для управления учетными записями используется Active Directory. Доступ к ресурсам, развернутым Azure Blueprint, защищен от повторных атак встроенными функциями Kerberos в Azure Active Directory, Active Directory и операционной системе Windows. При аутентификации Kerberos аутентификатор, отправляемый клиентом, содержит дополнительные данные, такие как зашифрованный список IP-адресов, метки времени клиента и время жизни билета. Если пакет повторяется, то проверяется метка времени. Если в метке времени указано более раннее время, чем в предыдущем аутентификаторе, или совпадающее с ним время, то пакет отклоняется. |
| **Поставщик&nbsp;(Microsoft&nbsp;Azure)** | Azure предоставляет службы для отслеживания удостоверения и интеграции с хранилищами удостоверений, которые вы, возможно, уже используете. Azure AD — это комплексная служба управления идентификацией и доступом для облака, которая помогает защищать доступ к данным в локальных и облачных приложениях. Azure AD также упрощает управление пользователями и группами, объединяя основные службы каталогов, расширенное управление идентификацией, безопасность и управление доступом к приложениям. |


 ## <a name="ncsc-cloud-security-principle-11"></a>Принцип облачной безопасности NCSC 11
### <a name="external-interface-protection"></a>Защита внешнего интерфейса
Все внешние или менее доверенные интерфейсы службы должны быть идентифицированы и надлежащим образом защищены.
Если некоторые из открытых интерфейсов являются частными (например, интерфейсы управления), то последствия компрометации могут быть более значительными.
Вы можете использовать разные модели для подключения к облачным службам, которые подвергают ваши корпоративные системы разным уровням риска.


**Обязанности:** `Shared`


|||
|---|---|
| **Клиент** | Azure Blueprint развертывает ресурсы в архитектуре с отдельной веб-подсетью, подсетью базы данных, подсетью Active Directory и подсетью управления. Подсети логически разделены правилами групп безопасности сети, применяемыми к отдельным подсетям, чтобы проходил только трафик, который необходим для функций системы и управления (например, внешний трафик не может получить доступ к базам данных, управлению или подсетям Active Directory).  <br /> <br /> Для управления внешними подключениями к развернутому клиентом веб-приложению развертывается шлюз приложения. Внешние подключения для доступа к управлению ограничены jumpbox (узлом-бастионом), развернутым в подсети управления, с правилами группы безопасности сети, применяемыми для принятия внешних подключений только с разрешенных IP-адресов. |
| **Поставщик&nbsp;(Microsoft&nbsp;Azure)** | Корпорация Майкрософт использует метод, который она называет Red Teaming, для улучшения контроля и процессов безопасности Azure посредством регулярного тестирования на проникновение в систему безопасности. Red Team — это группа штатных сотрудников Майкрософт, выполняющих целенаправленные и постоянные атаки служб инфраструктуры, платформ и приложений Майкрософт, а не приложений и данных клиентов. <br /> <br /> Дополнительные сведения о Red Teaming и Blue Teaming см. в полном описании [таблицы обязанностей клиентов в рамках Azure Blueprint и принципов облачной безопасности NCSC](https://aka.ms/blueprintuk-gcrm).  |


 ## <a name="ncsc-cloud-security-principle-12"></a>Принцип облачной безопасности NCSC 12
### <a name="secure-service-administration"></a>Безопасное администрирование служб
Системы, используемые для администрирования облачной службы, будут иметь строго привилегированный доступ к этой службе. Их компрометирование окажет значительное влияние, в том числе попытки обхода контроля безопасности и кражи или манипулирования большими объемами данных.
Проектирование, реализация и управление системами администрирования должно соответствовать рекомендациям компании с учетом их высокой ценности для злоумышленников.


**Обязанности:** `Shared`


|||
|---|---|
| **Клиент** | Клиент несет ответственность за обеспечение безопасной рабочей станции для администрирования своей подписки Azure и развернутых ресурсов (включая приложения, операционные системы, базы данных и программное обеспечение). В заявлении клиента о реализации средств управления должны быть рассмотрены механизмы, используемые для снижения риска взлома ресурсов, развернутых клиентами. |
| **Поставщик&nbsp;(Microsoft&nbsp;Azure)** | Эксплуатационные сотрудники Microsoft Azure должны использовать безопасные рабочие станции администрирования (SAW) или рабочие станции с привилегированным доступом (PAW). Подход с использованием безопасных рабочих станций администрирования — это уже известный, но более широкий подход, который заключается в том, что административный персонал должен использовать отдельные учетные записи администратора и пользователя. Здесь используется отдельно назначенная учетная запись администратора, отличная от учетной записи обычного пользователя. Это разделение лежит в основе подхода с использованием безопасных рабочих станций администрирования, в соответствии с которым для работы с конфиденциальными учетными записями предоставляется надежная рабочая станция. |


 ## <a name="ncsc-cloud-security-principle-13"></a>Принцип облачной безопасности NCSC 13
### <a name="audit-information-for-users"></a>Данные аудита для пользователей
Вам должны быть предоставлены аудиторские записи, необходимые для контроля доступа к вашей службе и содержащимся в ней данным. Тип доступных вам данных аудита будет иметь прямое влияние на вашу способность обнаруживать ненадлежащую или злонамеренную деятельность и реагировать на нее в разумные сроки.


**Обязанности:** `Shared`


|||
|---|---|
| **Клиент** | В рамках этого принципа Azure отслеживает события, собираемые журналами действий для развернутых ресурсов, журналами уровня ОС и журналами Active Directory. Эти журналы событий включают информацию, достаточную для определения того, когда происходят события, источника события, результата события и другой подробной информации, необходимой при расследовании инцидентов безопасности. Клиенты могут выбирать дополнительные события для аудита в зависимости от требований. Журналы аудита всех ресурсов Azure доступны на портале Azure. |
| **Поставщик&nbsp;(Microsoft&nbsp;Azure)** | Azure Log Analytics собирает записи о событиях, происходящих в системах и сетях организации, как только они появляются, прежде чем кто-либо сможет вмешаться в них, и позволяет проводить различные типы анализа, сопоставляя данные на нескольких компьютерах. Azure позволяет клиентам создавать и собирать события безопасности из ролей "инфраструктура как услуга" (IaaS) и "платформа как услуга" (PaaS) Azure в центральное хранилище своих подписок. Эти собранные события можно экспортировать в локальные системы управления сведениями о безопасности и событиями (SIEM) в целях непрерывного мониторинга. После передачи данных в хранилище существует много вариантов просмотра диагностических данных. <br /> <br /> Встроенная диагностика Azure может помочь с отладкой. Для приложений, развернутых в Azure, по умолчанию включен набор событий безопасности операционной системы. Клиенты могут добавлять, удалять или изменять события, аудит которых будет выполняться, путем настройки политики аудита ОС. <br /> <br /> На высоком уровне довольно легко и просто начать сбор журналов с помощью переадресации событий Windows (WEF) или более продвинутой системы диагностики Azure при развертывании виртуальных машин Windows с использованием IaaS в Azure. Кроме того, систему диагностики Azure можно настроить для сбора журналов и событий из экземпляров ролей PaaS. При использовании виртуальных машин на основе IaaS клиент просто настраивает и включает необходимые события безопасности так же, как позволяет серверам Windows регистрировать аудит в своем локальном центре обработки данных. Для веб-приложений также возможно включить ведение журнала IIS, если это основное приложение и развертывание в Azure. Клиенты всегда могут хранить данные безопасности в учетных записях в поддерживаемых географических расположениях по своему выбору для удовлетворения требований к независимости данных. |


 ## <a name="ncsc-cloud-security-principle-14"></a>Принцип облачной безопасности NCSC 14
### <a name="secure-use-of-the-service"></a>Безопасное использование службы
Безопасность облачных служб и данных, хранящихся в них, может оказаться под угрозой, если вы используете службу ненадлежащим образом. Следовательно, у вас будут определенные обязанности при использовании службы, чтобы ваши данные были надлежащим образом защищены.
Масштаб ответственности будет зависеть от моделей развертывания облачной службы и сценария, в котором вы собираетесь использовать эту службу. Определенные возможности отдельных служб также могут иметь значения. Например, то, как сеть доставки содержимого защищает ваш закрытый ключ, или то, как поставщик облачных платежей обнаруживает мошеннические транзакции, является важным с точки зрения безопасности, помимо общих принципов безопасности облачных служб.  
В случае предложений IaaS и PaaS вы отвечаете за важные аспекты безопасности ваших данных и рабочих нагрузок. Например, если вы получаете экземпляр вычислений IaaS, вы, как правило, несете ответственность за установку современной операционной системы, ее надежную настройку, надежное развертывание любых приложений, а также поддержку этого экземпляра путем применения исправлений или выполнения необходимого обслуживания.


**Обязанности:** `Customer`

> [!NOTE]
> Центр безопасности Azure позволяет клиентам предотвращать и выявлять угрозы, а также реагировать на них благодаря более полной информации о состоянии ресурсов Azure и повышенному контролю их безопасности Центр безопасности Azure предоставляет встроенные функции мониторинга безопасности и управления политиками для подписок Azure, помогает выявлять угрозы, которые в противном случае могли бы оказаться незамеченными, и взаимодействует с широким комплексом решений по обеспечению безопасности.
