# <a name="degradation-in-trace-severity-ratio-preview"></a>Снижение коэффициент важности трассировки (Предварительная версия)

Трассировки часто используются в приложениях, как они помогают понять все того, что происходит в фоновом. При неправильном, трассировки предоставляют решающее значение видимость последовательность событий, что приведет к нежелательным состояния. Хотя обычно неструктурированных трассировок, имеется единственное, что конкретно могут быть получены из них — уровень серьезности. В устойчивом состоянии приложения, ожидается отношение между трассировками «хорошие» (*сведения* и *Verbose*) и «плохое» трассировки (*предупреждение*, *ошибка* и *критическое*) будет оставаться стабильной. Предполагается, что «плохое» трассировки может произойти на регулярной основе в определенной степени из-за множеству других причин (временный сетевой проблемы для экземпляра). Но когда реальной проблемы начинается растет, он обычно манифесты как увеличение в относительные пропорции трассировок «хорошие» и «плохое» трассировок. Смарт-обнаружение аналитики приложений автоматически анализирует данные трассировки в журнал для приложения и может предупредить вас о нестандартных шаблонах в серьезность телеметрии трассировки.

Для этой функции требуется не специальных настроек, отличных от настройке ведения журнала трассировки для приложения (узнать, как настроить прослушиватель журнала трассировки для [.NET](https://docs.microsoft.com/azure/application-insights/app-insights-asp-net-trace-logs) или [Java](https://docs.microsoft.com/azure/application-insights/app-insights-java-trace-logs)). Он остается активным, если приложение формирует достаточно данных телеметрии исключений.

## <a name="when-would-i-get-this-type-of-smart-detection-notification"></a>Когда будет получить уведомления смарт-обнаружение такого типа
Этот тип уведомлений можно получить, если отношение между трассировками «хорошие» (трассировки в журнал с уровнем *сведения* или *Verbose*) и «плохое» трассировки (трассировки в журнал с уровнем *предупреждение*, * Ошибки, или *Неустранимая ошибка*) приводит к снижению в определенный день, сравниваемые с базовыми, вычисленное за предыдущую неделю.

## <a name="does-my-app-definitely-have-a-problem"></a>В работе моего приложения обязательно существует проблема?
Уведомление не означает, что в работе приложения возникла проблема. Несмотря на то, что снижение в отношение между «хорошие» и «плохое» трассировки может указывать на проблемы приложения, возможно, это изменение в отношение информационный характер. Например увеличение причиной может быть новый поток в приложении, создание дополнительных трассировки «плохое», чем существующих процессов).

## <a name="how-do-i-fix-it"></a>Как ее исправить?
Уведомления содержат диагностических сведений для поддержки в процессе диагностики:
1. **"Рассмотрение".** Уведомление показывает, сколько операций затрагиваются. Это поможет назначить приоритет проблемы.
2. **Область действия.** Проблема влияет на весь трафик или только некоторые операции? Эти сведения можно получить из уведомления.
3. **Выполните диагностику.** Можно использовать связанные элементы и отчеты, связывание с вспомогательные сведения, помогающие дальнейшей диагностики проблемы.
