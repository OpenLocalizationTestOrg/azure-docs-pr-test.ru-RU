## <a name="block-and-unblock-users"></a>Блокировать и разблокировать пользователей

Используйте _блокировать и разблокировать пользователей_ функции, чтобы предотвратить получение запросов проверки подлинности пользователей. Любые попытки выполнить аутентификацию заблокированных пользователей отклоняются автоматически. Пользователи остаются заблокированными 90 дней с момента они блокируются.

### <a name="block-a-user"></a>Блокировка пользователя

1. Войдите на [портал Azure](https://portal.azure.com) с использованием учетной записи администратора.
2. Перейдите к **Azure Active Directory** > **многофакторной проверки Подлинности сервера** > **блокировка и разблокировка пользователей**.
3. Выберите **добавить** Чтобы заблокировать пользователя.
4. Выберите **группы репликации**. Введите имя пользователя для заблокированных пользователей как **username<span></span>@domain.com**. Введите комментарий в **Причина** поля.
5. Выберите **добавить** окончания блокировки пользователя.

### <a name="unblock-a-user"></a>Разблокирование пользователя

1. Войдите на [портал Azure](https://portal.azure.com) с использованием учетной записи администратора.
2. Перейдите к **Azure Active Directory** > **многофакторной проверки Подлинности сервера** > **блокировка и разблокировка пользователей**.
3. Выберите **Unblock** в **действия** столбец рядом с пользователем для разблокирования.
4. Введите комментарий в поле **Причина разблокировки**.
5. Выберите **Unblock** завершения разблокирования пользователя.

## <a name="fraud-alert"></a>Предупреждение о мошенничестве

Настройка _предупреждение о мошенничестве_ , которая отчетов мошеннической попытки доступа к ресурсам вашей пользователей. Пользователей можно сообщить о мошенничестве попыток с помощью мобильного приложения или с помощью телефона.

### <a name="turn-on-fraud-alerts"></a>Включить предупреждения о мошенничестве

1. Войдите на [портал Azure](https://portal.azure.com) с использованием учетной записи администратора.
2. Перейдите к **Azure Active Directory** > **многофакторной проверки Подлинности сервера** > **предупреждение о мошенничестве**.

   ![Включить предупреждения о мошенничестве](./media/multi-factor-authentication-whats-next/fraudalert.png)

3. Задать **разрешить пользователям отправлять предупреждения о мошенничестве** параметру **на**.
4. Щелкните **Сохранить**.

### <a name="configuration-options"></a>Варианты настройки

- **Блокировать пользователя при получении мошенничестве**: Если пользователь сообщает о мошенничестве, их учетная запись заблокирована 90 дней или пока администратор разблокирования учетной записи. Администратор можно просмотреть входа в систему с помощью отчета вход и предпринять соответствующие действия для предотвращения будущих мошенничества. Затем администратор может [разблокировать](#unblock-a-user) учетную запись пользователя.
- **Код, чтобы сообщить о мошенничестве во время начального приветствия**: пользователи получают телефонный звонок для выполнения двухшаговую проверку, они обычно нажмите  **#**  для подтверждения вход. Чтобы сообщить о мошенничестве пользователь вводит код перед нажатие  **#** . По умолчанию используется код **0**, но вы можете его изменить.

  >[!NOTE]
  >По умолчанию голосовые приветствия корпорации Майкрософт поручить пользователям клавишу **0#** Отправить предупреждение о мошенничестве. Если вы хотите использовать код, отличный от **0**, записи и отправить свои собственные настраиваемые голосовые приветствия с соответствующие инструкции для пользователей.
  >

### <a name="view-fraud-reports"></a>Просмотр отчетов о мошенничестве

1. Войдите на [портале Azure](https://portal.azure.com).
2. Выберите **Azure Active Directory** > **входа в систему**. Отчет о мошенничестве теперь является частью стандартного отчета Azure AD войти в систему.

## <a name="one-time-bypass"></a>Разовый обход

_Одноразовый обход проверки_ функция позволяет пользователю однократную проверку подлинности без выполнения двухшаговую проверку. Обход проверки является временным процессом и истекает через заданное количество секунд. В ситуациях, где мобильного приложения или телефон не получает уведомлений или телефонный звонок поэтому пользователь может получить доступ к нужный ресурс можно разрешить одноразовый обход проверки.

### <a name="create-a-one-time-bypass"></a>Создание разового обхода проверки

1. Войдите на [портал Azure](https://portal.azure.com) с использованием учетной записи администратора.
2. Перейдите к **Azure Active Directory** > **многофакторной проверки Подлинности сервера** > **одноразовый обход проверки**.

   ![Создание одноразового обхода проверки](./media/multi-factor-authentication-whats-next/onetimebypass.png)

3. Выберите **Добавить**.
4. При необходимости выберите группу репликации для обхода.
5. Введите имя пользователя как **username<span></span>@domain.com**. Введите количество секунд, которые должен длиться обход проверки. Введите причину для обхода. 
6. Выберите **Добавить**. Ограничение времени вступает в силу немедленно. Пользователь должен выполнить вход до истечения срока действия одноразового обхода проверки. 

### <a name="view-the-one-time-bypass-report"></a>Просмотр отчета о разовом обходе проверки

1. Войдите на [портале Azure](https://portal.azure.com).
2. Перейдите к **Active Directory** > **многофакторной проверки Подлинности сервера** > **одноразовый обход проверки**.

## <a name="custom-voice-messages"></a>Пользовательские голосовые сообщения
Можно использовать собственные записи или greetings для двухшаговой проверки с _настраиваемых голосовых сообщений_ компонентов. Эти сообщения можно использовать в дополнение к или заменить записи Майкрософт.

Прежде чем начать, необходимо учитывать следующие ограничения.

* Поддерживаются только форматы файлов WAV и MP3.
* Максимальный размер файла составляет 5 МБ.
* Сообщения о проверке подлинности должны длиться меньше, чем 20 секунд. Сообщения, которые являются более 20 секунд может привести к сбою проверки. Пользователь может не отвечать до конца сообщения, и срок проверки истечет.

### <a name="set-up-a-custom-message"></a>Настройка пользовательского сообщения

1. Войдите на [портал Azure](https://portal.azure.com) с использованием учетной записи администратора.
2. Перейдите к **Azure Active Directory** > **многофакторной проверки Подлинности сервера** > **параметры телефонный звонок**.

   ![Записи phone пользовательских сообщений](./media/multi-factor-authentication-whats-next/phonecallsettings.png)

3. Щелкните **Добавить приветствие**.
4. Выберите тип приветствия. Выберите язык.
5. Выберите звуковой файл в формате MP3 или WAV для отправки.
6. Выберите **Добавить**.

## <a name="caching-in-azure-multi-factor-authentication"></a>Кэширование в службе Многофакторной идентификации Azure

Можно задать период времени, чтобы разрешить попыток проверки подлинности после проверки подлинности пользователя с помощью _кэширование_ компонентов. Последующие попытки проверки подлинности для пользователя в течение указанного периода времени успешно автоматически. Кэширование в основном используется, когда локальных систем, таких как VPN, отправляют несколько запросов проверки по-прежнему во время первого запроса. Эта функция позволяет последующие запросы для успешного выполнения автоматически, после первого идет проверка успешного пользователя. 

>[!NOTE]
>Возможность кэширования не предназначен для использования во время входа в Azure Active Directory (Azure AD).

### <a name="set-up-caching"></a>Настройка кэширования 

1. Войдите на [портал Azure](https://portal.azure.com) с использованием учетной записи администратора.
2. Перейдите к **Azure Active Directory** > **многофакторной проверки Подлинности сервера** > **правила кэширования**.

   ![Настройка правила кэширования](./media/multi-factor-authentication-whats-next/cachingrules.png)

3. Выберите **Добавить**.
4. Выберите **тип кэша** из раскрывающегося списка. Введите максимальное число **кэшировать секунд**. 
5. При необходимости выберите тип проверки подлинности и указать приложение. 
6. Выберите **Добавить**.


## <a name="trusted-ips"></a>Надежные IP-адреса

_Надежных IP-адресов_ функция многофакторной проверки подлинности Azure используется администраторами управляемого или федеративного клиента. Функция пропускает двухшаговую проверку для пользователи, выполнившие вход интрасети компании. Компонент доступен в полной версии многофакторной проверки подлинности Azure, а не бесплатная версия для администраторов. Дополнительные сведения о том, как получить полную версию службы Многофакторной идентификации Azure, см. в [этой статье](multi-factor-authentication.md).

| Тип клиента Azure AD | Параметры функций надежных IP-адресов |
|:--- |:--- |
| Управляемые |**Определенного диапазона IP-адресов**: Администраторы указать диапазон IP-адресов, которые могут обойти двухшаговую проверку для пользователи, выполнившие вход интрасети компании.|
| Федеративные |**Все федеративные пользователи**: все федеративные пользователи, выполнившие вход из внутри организации могут обходить двухшаговую проверку. Пользователям обходить проверку с помощью утверждения, выданный служб федерации Active Directory (AD FS).<br/>**Определенного диапазона IP-адресов**: Администраторы указать диапазон IP-адресов, которые могут обойти двухшаговую проверку для пользователи, выполнившие вход интрасети компании. |

Надежные IP-адреса обход работает только с внутри интрасети компании. При выборе **все федеративные пользователи** параметр и пользователь входит с за пределами интрасети компании, пользователь должен проходить проверку подлинности, с помощью двухэтапной проверки. Процесс одинаково даже если пользователь не предоставит утверждений AD FS. 

**Взаимодействие с пользователем внутри корпоративной сети**

При отключении функции надежных IP-адресов, двухшаговую проверку требуется для браузерных потоков. Пароли приложений требуются для более старых многофункциональных клиентских приложений. 

Если включена функция надежных IP-адресов, двухшаговая проверка — *не* требуется для браузерных потоков. Пароли приложений — *не* требуется для более старых насыщенных клиентских приложений, при условии, что пользователь еще не создан пароль приложения. Когда пароль приложения уже используется, пароль не требуется. 

**Взаимодействие с пользователем за пределами корпоративной сети**

Независимо от того, включена ли функция надежных IP-адресов двухшаговую проверку требуется для браузерных потоков. Пароли приложений требуются для более старых многофункциональных клиентских приложений. 

### <a name="enable-named-locations-by-using-conditional-access"></a>Включить именованных расположений с помощью условного доступа

1. Войдите на [портале Azure](https://portal.azure.com).
2. В левой части экрана выберите **Azure Active Directory** > **условного доступа** > **расположения с именем**.
3. Выберите **новое расположение**.
4. Введите имя для расположения.
5. Выберите **пометить как надежное расположение**.
6. Введите диапазон IP-адресов в нотации CIDR, например **192.168.1.1/24**.
7. Нажмите кнопку **Создать**.

### <a name="enable-the-trusted-ips-feature-by-using-conditional-access"></a>Включить функцию надежных IP-адресов с помощью условного доступа

1. Войдите на [портале Azure](https://portal.azure.com).
2. В левой части экрана выберите **Azure Active Directory** > **условного доступа** > **расположения с именем**.
3. Выберите **Настройка многофакторной проверки Подлинности надежных IP-адресов**.
4. На **параметры службы** в разделе **надежных IP-адресов**, выберите один из следующих двух параметров:
   
   * **Для запросов от федеративных пользователей из моей интрасети**: чтобы выбрать этот параметр, установите флажок. Все федеративные пользователи, входа из корпоративной сети обхода двухшаговую проверку, используя утверждения, выданный AD FS. Убедитесь, что в службах AD FS установлено правило для добавления утверждение интрасети для соответствующего трафика. Если правило не существует, создайте следующее правило в AD FS:<br/>

     ```
     c:[Type== "http://schemas.microsoft.com/ws/2012/01/insidecorporatenetwork"] => issue(claim = c);
     ```
     
   * **Для запросов из определенного диапазона общедоступных IP-адресов**: чтобы выбрать этот параметр, введите IP-адрес в текстовом поле, используя нотацию CIDR.
   
     * Для IP-адресов, которые находятся в списке xxx.xxx.xxx.1 через xxx.xxx.xxx.254, нотация как **xxx.xxx.xxx.0/24**.
     * Один IP-адрес, используют нотацию как **xxx<span></span>.xxx.xxx.xxx/32**.
     
     Введите до 50 диапазоны IP-адресов. Пользователи, выполняющие вход с этих IP-адресов, обходят двухфакторную проверку подлинности.

5. Щелкните **Сохранить**.

### <a name="enable-the-trusted-ips-feature-by-using-service-settings"></a>Включить функцию надежных IP-адресов с помощью параметров службы

1. Войдите на [портале Azure](https://portal.azure.com).
2. Слева выберите **Azure Active Directory** > **Пользователи и группы** > **Все пользователи**.
3. Выберите **Многофакторная идентификация**.
4. В разделе "Многофакторная аутентификация" выберите **Параметры службы**.
5. На **параметры службы** в разделе **надежных IP-адресов**, выберите один из следующих двух параметров:
   
   * **Для запросов от федеративных пользователей из моей интрасети**: чтобы выбрать этот параметр, установите флажок. Все федеративные пользователи, входа из корпоративной сети обхода двухшаговую проверку, используя утверждения, выданный AD FS. Убедитесь, что в службах AD FS установлено правило для добавления утверждение интрасети для соответствующего трафика. Если правило не существует, создайте следующее правило в AD FS:<br/>

     ```
     c:[Type== "http://schemas.microsoft.com/ws/2012/01/insidecorporatenetwork"] => issue(claim = c);
     ```

   * **Для запросов из указанного диапазона IP-адресов подсети**: чтобы выбрать этот параметр, введите IP-адрес в текстовом поле, используя нотацию CIDR. 
     
     * Для IP-адресов, которые находятся в списке xxx.xxx.xxx.1 через xxx.xxx.xxx.254, нотация как **xxx.xxx.xxx.0/24**.
     * Один IP-адрес, используют нотацию как **xxx<span></span>.xxx.xxx.xxx/32**.
     
     Введите до 50 диапазоны IP-адресов. Пользователи, выполняющие вход с этих IP-адресов, обходят двухфакторную проверку подлинности.

6. Щелкните **Сохранить**.

## <a name="app-passwords"></a>Пароли приложений

Некоторые приложения, например Office 2010 или более ранней версии и Apple Mail не поддерживают двухшаговую проверку. Приложения не настроен на прием второй проверки. Чтобы использовать эти приложения, необходимо воспользоваться преимуществами _пароли приложений_ компонентов. Пароль приложения вместо традиционных пароль позволяет разрешить приложению обхода двухшаговую проверку и продолжить работу.

>[!NOTE]
>Современная проверка подлинности для клиентов Microsoft Office 2013 и более поздних версий
> 
>Клиенты Office 2013 и более поздней версии (включая Outlook), поддерживают протоколы современную проверку подлинности, их можно включить для работы с помощью двухэтапной проверки. После включения клиента пароли приложений не являются обязательными для клиента. Дополнительные сведения см. в записи блога [Office 2013 modern authentication public preview announced](https://blogs.office.com/2015/03/23/office-2013-modern-authentication-public-preview-announced/) (Анонсирована общедоступная предварительная версия современной службы многофакторной идентификации для Office 2013).
>

### <a name="considerations-about-app-passwords"></a>Вопросы о паролях приложений

При использовании паролей приложений, необходимо учитывайте следующие важные моменты.

* Пароли приложений вводятся только один раз на каждое приложение. Пользователям не нужно отслеживания пароли или вводить их каждый раз.
* В действительности пароль создается автоматически, а не задается пользователем. Автоматически созданный пароль сложнее подобрать, и он более безопасен.
* Максимальное количество паролей на пользователя — 40. 
* Приложения, которые кэшируют пароли и использовать их в локальных сценариях можно запустить ошибкой, поскольку пароль приложения не известен за пределами рабочей или учебной учетной записи. Пример такого сценария — сообщения электронной почты Exchange, которые являются локальными, однако архивация почты выполняется в облаке. В этом случае пароль не работает.
* После включения многофакторной проверки подлинности в учетной записи пользователя пароли приложений можно использовать с наиболее внебраузерных клиентов, таких как Outlook и Microsoft Скайп для бизнеса. Административные действия нельзя выполнить с помощью паролей приложений приложения без браузера, такие как Windows PowerShell. Невозможно выполнить действия, даже в том случае, когда у пользователя есть учетная запись администратора. Для запуска скриптов PowerShell, создать надежный пароль учетной записи службы и не включайте учетную запись для двухшаговой проверки.

>[!WARNING]
>Пароли приложений не работают в гибридных средах, где клиенты взаимодействуют как с локальными, так и с облачными конечными точками автообнаружения. Пароли домена необходимы для проверки подлинности в локальной среде. Пароли приложений требуются для проверки подлинности в облаке.
>

### <a name="guidance-for-app-password-names"></a>Руководство по имена паролей приложений

Имена паролей приложений соответствовали устройству, на котором они используются. Если у вас есть ноутбук с приложениями без браузера, например Outlook, Word и Excel, создать один пароль приложения с именем **ноутбук** для этих приложений. Создать другой пароль приложения с именем **Desktop** для тех же приложений, работающих на настольном компьютере. 

>[!NOTE]
>Рекомендуется создать пароль одного приложения на устройство, а не пароль одно приложение на каждое приложение.

### <a name="federated-or-single-sign-on-app-passwords"></a>Пароли федеративного единого входа для приложения или

Azure AD поддерживает федерации или единого входа (SSO) с локальными Windows Server службами Active Directory домена (AD DS). Если в организации настроена федерация с Azure AD, и вы используете многофакторную проверку подлинности Azure, необходимо учитывать следующее о паролях приложений.

>[!NOTE]
>Следующие пункты относятся только к федеративным (SSO) клиентам.

* Пароли приложений проверяются Azure AD и поэтому обход федерации. Федерация активно используется только в том случае, если настройка паролей приложений.
* Для федеративных пользователей (SSO), в отличие от пассивного потока не связаться с поставщиком удостоверений (IdP). Пароли приложений хранятся в рабочую или учебную учетную запись. Если пользователь увольняется из компании, данные пользователя передаются рабочей или учебной учетной записи с помощью **DirSync** в режиме реального времени. Отключения/удаления учетной записи может занять до трех часов для синхронизации, который может задержать отключения/удаления пароля приложения в Azure AD.
* С помощью функции пароли приложений не принимаются параметры управления доступом в локальной среде клиента.
* Без проверки подлинности в локальной среде ведение журнала и аудит возможность доступна для использования с функцией пароли приложений.
* Некоторые расширенные архитектуры требуется сочетание учетные данные для двухшаговой проверки с клиентами. Эти учетные данные могут включать рабочего или учебного заведения имя пользователя учетной записи и пароли и пароли приложений. Требования зависят от того, как выполняется проверка подлинности. Для клиентов, проходящие проверку подлинности в локальной инфраструктуре рабочей или учебной учетной записи пользователя и пароль обязательный. Клиенты, проходящие проверку подлинности Azure AD пароль приложения не требуется.

  Например предположим, что у вас есть следующие архитектуры:

  * Локальный экземпляр Active Directory включена в федерацию с Azure AD.
  * Вы используете Exchange online.
  * Вы используете Скайп для бизнеса.
  * Вы используете многофакторную проверку подлинности Azure.

  ![С помощью паролей приложений в федеративной организации](./media/multi-factor-authentication-whats-next/federated.png)

  В этом случае можно использовать следующие учетные данные:

  * Для входа в Скайп для бизнеса, используйте имя рабочего или учебного заведения учетной записи пользователя и пароль.
  * Для получения доступа к адресной книге клиента Outlook, подключается к Exchange онлайн, используйте пароль приложения.

## <a name="remember-multi-factor-authentication-for-trusted-devices"></a>Запомнить многофакторной проверки подлинности для доверенных устройств
_Запомнить многофакторной проверки подлинности_ компонент для устройств и браузеров, которые являются доверенными пользователя является бесплатной для всех пользователей многофакторной проверки подлинности. Заданное число дней, после они успешно выполнившего вход на устройство с помощью многофакторной проверки подлинности пользователей, можно обойти последующие попытки проверки. Средство повышает удобство использования, сводя к минимуму количество раз, пользователь должен выполнить двухшаговую проверку на одном устройстве.

>[!IMPORTANT]
>При компрометации учетной записи или устройства запоминание многофакторной проверки подлинности для доверенных устройств может повлиять на безопасность. В случае потери доверенного устройства или компрометации учетной записи организации следует [восстановить Многофакторную идентификацию на всех устройствах](multi-factor-authentication-manage-users-and-devices.md#restore-mfa-on-all-remembered-devices-for-a-user).
>
>Операция восстановления отменяет доверенных состояние из всех устройств и пользователя, необходимое для выполнения двухшаговую проверку еще раз. Можно также Сообщите пользователям о необходимости восстановить многофакторную проверку подлинности на своих устройствах с инструкциями в [управления параметрами для двухшаговой проверки](./end-user/multi-factor-authentication-end-user-manage-settings.md#require-two-step-verification-again-on-a-device-youve-marked-as-trusted).
>

### <a name="how-the-feature-works"></a>Том, как работает

Запомнить многофакторной проверки подлинности наборы функций постоянный файл cookie в браузере, когда пользователь выбирает **не спрашивать в течение X дней** параметр при входе в систему. У пользователя не запрашиваются снова многофакторной проверки подлинности из этого же браузера до истечения срока действия файла cookie. Если пользователь открывает другой браузер на одном устройстве, или удаляет их файлов cookie, они предложено повторно проверить. 

**Не спрашивать в течение X дней** вариант не отображается для приложения без браузера, независимо от того, поддерживает ли приложение современную проверку подлинности. Эти приложения используют _токенов обновления_ новых токенов доступа, которые предоставляют каждый час. После проверки токена обновления Azure AD проверяет произошедшее последнего двухшаговую проверку в течение указанного числа дней. 

Функция сокращает количество проверок подлинности на веб-приложений, которые обычно запрос каждый раз. Компонент увеличит количество проверок подлинности для современной проверки подлинности клиентов, которые обычно запрашивают каждые 90 дней.

>[!IMPORTANT]
>**Запомнить многофакторной проверки подлинности** компонент несовместим с **оставаться в системе** компонент AD FS, когда пользователи выполнять двухшаговую проверку для службы федерации Active Directory через Azure многофакторной Проверка подлинности сервера или решения сторонних многофакторной проверки подлинности.
>
>Если пользователи выбрали **оставаться в системе** на AD FS, а также пометить устройства как доверенный для многофакторной проверки подлинности, пользователь не проверены выполнять действия автоматически, после **запомнить многофакторной проверки подлинности**количество дней срока действия. Azure AD запрашивает новую двухшаговую проверку, но службы федерации Active Directory возвращает маркер с исходной утверждения многофакторной проверки подлинности и даты, а не выполнение двухшаговую проверку еще раз. Подобная реакция отключает циклическую проверку между Azure AD и AD FS.
>
