# <a name="tutorial-process-edifact-invoices-using-azure-biztalk-services"></a>Руководство по обработке счетов EDIFACT с помощью служб BizTalk Azure

> [!INCLUDE [BizTalk Services is being retired, and replaced with Azure Logic Apps](../../includes/biztalk-services-retirement.md)]

Портал служб BizTalk можно использовать для настройки и развертывания соглашений EDIFACT и X12. В этом руководстве описывается создание соглашения EDIFACT для обмена счетами между торговыми партнерами. В качестве примера здесь используется комплексное бизнес-решение с двумя торговыми партнерами, компаниями Contoso и Northwind, которые обмениваются сообщениями EDIFACT.  

## <a name="sample-based-on-this-tutorial"></a>Пример, который используется в этом руководстве
В этом руководстве используется пример решения **Отправка счетов EDIFACT с помощью служб BizTalk**, который можно загрузить из [коллекции кодов MSDN](http://go.microsoft.com/fwlink/?LinkId=401005). Работа над руководством с использованием этого примера поможет вам разобраться в том, как построено это решение. Также с помощью этого руководства вы можете создать собственное решение. Это и есть основная задача руководства — объяснить, как построено это решение. Кроме того, руководство (насколько это возможно) согласовано с примером: в нем используются те же имена артефактов (схем, преобразований и т. д.), что и в примере.  

> [!NOTE]
> Так как в этом решении выполняется передача сообщения между мостами EAI и EDI, будет повторно использован пример [цепочки мостов служб BizTalk](http://code.msdn.microsoft.com/BizTalk-Bridge-chaining-2246b104) .  
> 
> 

## <a name="what-does-the-solution-do"></a>Каково предназначение этого решения?
В рамках этого решения компания Northwind получает счета EDIFACT от компании Contoso. Эти счета передаются не в стандартном формате EDI. Поэтому перед отправкой в компанию Northwind счет необходимо преобразовать в документ EDIFACT (также называемый INVOIC). Получив счет EDIFACT, компания Northwind должна обработать его и отправить компании Contoso контрольное сообщение (также называемое CONTRL).

![][1]  

Компания Contoso осуществляет этот бизнес-сценарий с помощью служб BizTalk Microsoft Azure.

* Contoso использует мосты EAI для преобразования исходного счета в формат EDIFACT INVOIC.
* Затем мост EAI отправляет сообщение в мост отправки EDI, который развернут в рамках соглашения, настроенного на портале служб BizTalk.
* Мост отправки EDI обрабатывает счет EDIFACT в формате INVOIC и перенаправляет его компании Northwind.
* После получения счета Northwind возвращает сообщение CONTRL в мост получения EDI, который также развернут в рамках соглашения.  

> [!NOTE]
> При необходимости в этом решении также может использоваться пакетная обработка (для отправки счетов в составе пакетов, а не по отдельности).  
> 
> 

В этом руководстве мы воспользуемся очередями служебной шины для отправки счета из Contoso в Northwind, а также для получения подтверждения от Northwind. Эти очереди могут быть созданы с помощью клиентского приложения, которое доступно для загрузки и включено в пакет примера, используемого в этом руководстве.

Предположим, вы уже подготовили пространство имен служебной шины с именем **edifactbts**.

* Зарегистрируйте ее на портале служб BizTalk. Ознакомьтесь со статьей [Регистрация и обновление развертывания службы BizTalk на портале служб BizTalk](https://msdn.microsoft.com/library/hh689837.aspx).
* Необходимо установить Visual Studio.
* Необходимо установить пакет SDK для служб BizTalk. Пакет SDK можно загрузить отсюда: [http://go.microsoft.com/fwlink/?LinkId=235057](http://go.microsoft.com/fwlink/?LinkId=235057)  

## <a name="step-1-create-the-service-bus-queues"></a>Шаг 1. Создание очередей служебной шины
В этом решении очереди служебной шины используются для обмена сообщениями между торговыми партнерами. Компании Contoso и Northwind отправляют сообщения в очереди, откуда их получают мосты EAI и EDI. Для этого решения требуется три очереди служебной шины:

* **northwindreceive** — компания Northwind получает счет от компании Contoso через эту очередь.
* **contosoreceive** — компания Contoso получает подтверждение от компании Northwind через эту очередь.
* **suspended** — все приостановленные сообщения направляются в эту очередь. Сообщения приостанавливаются, если во время их обработки происходит ошибка.

Эти очереди служебной шины можно создать с помощью клиентского приложения, включенного в пакет примера.  

1. В расположении, куда вы загрузили пример, откройте файл **Tutorial Sending Invoices Using BizTalk Services EDI Bridges.sln**.
2. Нажмите клавишу **F5**, чтобы выполнить сборку и запуск приложения **Tutorial Client**.
3. На экране укажите для служебной шины сведения ACS: пространство имен, имя издателя и ключ издателя.
   
   ![][2]  
4. Появится сообщение с предложением создать в пространстве имен служебной шины три очереди. Последовательно выберите **ОК**.
5. Оставьте приложение Tutorial Client запущенным. Щелкните **Служебная шина** > ***ваше пространство имен служебной шины*** > **Очереди** и убедитесь, что созданы три очереди.  

## <a name="step-2-create-and-deploy-trading-partner-agreement"></a>Шаг 2. Создание и развертывание соглашения между торговыми партнерами
Создайте соглашение между торговыми партнерами Contoso и Northwind. Это соглашение определяет торговые контракты между двумя партнерами (например, используемую схему обмена сообщениями, протокол обмена и т. д.). Соглашение между торговыми партнерами включает два моста EDI: один для отправки сообщений торговым партнерам (**мост отправки EDI**) и один для получения сообщений от торговых партнеров (**мост получения EDI**).

В рамках этого решения мост отправки EDI соответствует отправляющей стороне соглашения и используется для отправки счета EDIFACT из Contoso в Northwind. Подобным образом мост получения EDI соответствует принимающей стороне соглашения и используется для получения подтверждений от Northwind.  

### <a name="create-the-trading-partners"></a>Создание торговых партнеров
Начнем с создания торговых партнеров, представляющих компании Contoso и Northwind.  

1. На портале служб BizTalk на вкладке **Партнеры** щелкните **Добавить**.
2. На странице создания партнера введите **Contoso** в качестве имени партнера, а затем нажмите кнопку **Сохранить**.
3. Повторите этот шаг, чтобы создать второго партнера с именем **Northwind**.  

### <a name="create-the-agreement"></a>Создание соглашения
Соглашения между торговыми партнерами — это соглашения между бизнес-профилями торговых партнеров. Это решение использует профили по умолчанию, создаваемые автоматически при создании партнеров.  

1. На портале служб BizTalk щелкните **Соглашения** > **Добавить**.
2. На странице **Общие параметры** нового соглашения укажите значения, как показано на рисунке ниже, а затем нажмите кнопку **Продолжить**.
   
   ![][3]  
   
   После нажатия кнопки **Продолжить** станут доступными две вкладки: **Настройки получения** и **Настройки отправки**.
3. Создайте соглашение отправки между компаниями Contoso и Northwind. Это соглашение определяет способ отправки счета EDIFACT из Contoso в Northwind.
   
   1. Щелкните **Параметры отправки**.
   2. Оставьте значения по умолчанию на вкладках **Inbound URL** (Входящий URL-адрес), **Transform** (Преобразование) и **Пакетная обработка**.
   3. На вкладке **Протокол** в разделе **Схемы** передайте схему **EFACT_D93A_INVOIC.xsd**. Эта схема включена в пакет образца.
      
      ![][4]  
   4. На вкладке **Транспорт** укажите сведения об очередях служебной шины. В рамках соглашения отправляющая сторона будет использовать очередь **northwindreceive** для отправки счета EDIFACT в Northwind и очередь **suspended** для перенаправления сообщений, приостановленных из-за ошибки во время обработки. Вы создали эти очереди в разделе **Шаг 1. Создание очередей служебной шины** .
      
      ![][5]  
      
      В разделе **Параметры транспорта > Тип транспорта** и **Message Suspension Settings (Параметры приостановки сообщения) > Тип транспорта** выберите служебную шину Azure и укажите значения, как показано на рисунке.
4. Создайте соглашение получения между компаниями Contoso и Northwind. Это соглашение определяет способ получения подтверждения компанией Contoso от Northwind.
   
   1. Щелкните **Параметры получения**.
   2. Оставьте значения по умолчанию на вкладках **Транспорт** и **Transform** (Преобразование).
   3. На вкладке **Протокол** в разделе **Схемы** передайте схему **EFACT_4.1_CONTRL.xsd**. Эта схема включена в пакет образца.
   4. На вкладке **Маршрут** создайте фильтр, чтобы в Contoso направлялись только подтверждения от Northwind. В разделе **Route Settings** (Параметры маршрутизации) нажмите кнопку **Добавить**, чтобы создать фильтр маршрутизации.
      
      ![][6]  
      
      1. Укажите значения для параметров **Имя правила**, **Route rule** (Правило маршрутизации) и **Route destination** (Назначение маршрута), как показано на рисунке.
      2. Выберите команду **Сохранить**.
   5. Вернитесь на вкладку **Маршрут** и укажите, куда будут направляться приостановленные подтверждения (во время обработки которых произошла ошибка). Укажите следующие значения параметров: для типа транспорта — "Служебная шина Azure", для типа назначения маршрута — **Очередь**, для типа аутентификации — **Подписанный URL-адрес** (SAS). Укажите строку подключения SAS для пространства имен служебной шины и введите имя очереди **suspended**.
5. В завершение нажмите кнопку **Развернуть** , чтобы развернуть соглашение. Обратите внимание на конечные точки, в которых развертываются соглашения отправки и получения.
   
   * Запишите значение конечной точки на вкладке **Настройки отправки** в разделе **Inbound URL** (Входящий URL-адрес). Отправить сообщение из Contoso в Northwind с помощью моста отправки EDI можно только через эту конечную точку.
   * Запишите значение конечной точки на вкладке **Настройки получения** в разделе **Транспорт**. Отправить сообщение из Northwind в Contoso с помощью моста получения EDI можно только через эту конечную точку.  

## <a name="step-3-create-and-deploy-the-biztalk-services-project"></a>Шаг 3. Создание и развертывание проекта служб BizTalk
На предыдущем шаге вы развернули соглашения отправки и получения EDI для обработки подтверждений и счетов EDIFACT. Эти соглашения могут обрабатывать только те сообщения, которые соответствуют стандартной схеме сообщений EDIFACT. Но в рамках нашего сценария компания Contoso отправляет счет в Northwind, используя корпоративную (собственную) схему. Поэтому, прежде чем сообщение будет отправлено в мост отправки EDI, эту корпоративную схему необходимо преобразовать в стандартную схему счета EDIFACT. Это можно сделать с помощью проекта EAI служб BizTalk.

Проект **InvoiceProcessingBridge**служб BizTalk, который выполняет преобразования сообщений, также включен в загруженный пакет примера. Проект включает следующие артефакты:

* **INHOUSEINVOICE.XSD** — схема корпоративного счета, отправляемого в Northwind.
* **EFACT_D93A_INVOIC.XSD** — схема стандартного счета EDIFACT.
* **EFACT_4.1_CONTRL.XSD** — схема подтверждения EDIFACT, которое Northwind отправляет в Contoso.
* **INHOUSEINVOICE_to_D93AINVOIC.TRFM** — преобразование, сопоставляющее схему корпоративного счета со схемой стандартного счета EDIFACT.  

### <a name="create-the-biztalk-services-project"></a>Создание проекта служб BizTalk
1. В решении Visual Studio разверните проект InvoiceProcessingBridge и откройте файл **MessageFlowItinerary.bcs** .
2. Щелкните в любом месте холста и введите в окне свойств **URL-адрес службы BizTalk** , чтобы указать имя подписки служб BizTalk. Например, `https://contosowabs.biztalk.windows.net`.
   
   ![][7]  
3. С панели элементов перетащите на холст **односторонний мост XML** . Задайте для свойств моста **Имя сущности** и **Относительный адрес** значение **ProcessInvoiceBridge**. Дважды щелкните **ProcessInvoiceBridge** , чтобы открыть поверхность конфигурации моста.
4. В поле **Типы сообщений** нажмите кнопку **+**, чтобы указать схему входящего сообщения. Схемой входящего сообщения для моста EAI всегда будет корпоративный счет, поэтому выберите значение **INHOUSEINVOICE**.
   
   ![][8]  
5. Щелкните фигуру **Преобразование XML** и в окне свойств для свойства **Карты** нажмите кнопку с многоточием (**…**). В диалоговом окне **Maps Selection** (Выбор сопоставлений) выберите файл преобразования **INHOUSEINVOICE_to_D93AINVOIC** и нажмите кнопку **ОК**.
   
   ![][9]  
6. Вернитесь к файлу **MessageFlowItinerary.bcs** и перетащите с панели элементов элемент **Two-Way External Service Endpoint** (Двусторонняя внешняя конечная точка службы) вправо от **ProcessInvoiceBridge**. Задайте для свойства **Имя сущности** значение **EDIBridge**.
7. В обозревателе решений разверните файл **MessageFlowItinerary.bcs** и дважды щелкните файл **EDIBridge.config**. Замените содержимое файла **EDIBridge.config** следующим кодом.
   
   > [!NOTE]
   > Зачем редактировать файл конфигурации? Внешняя конечная точка службы, которую мы добавили на холст конструктора мостов, представляет мосты EDI, развернутые ранее. Мосты EDI являются двусторонними, то есть обеспечивают отправку и получение. Однако мост EAI, который мы добавили в конструктор, является односторонним. Поэтому для обработки разных шаблонов обмена сообщениями двух мостов мы используем настраиваемое поведение моста, представленное настройками в файле конфигурации. Настраиваемое поведение также осуществляет аутентификацию на конечной точке моста отправки EDI. Настраиваемое поведение доступно как отдельный пример на странице [Пример цепочки мостов служб BizTalk — EAI и EDI](http://code.msdn.microsoft.com/BizTalk-Bridge-chaining-2246b104). Этот пример используется в нашем решении.  
   > 
   > 
   
   ```
   <?xml version="1.0" encoding="utf-8"?>
   <configuration>
     <system.serviceModel>
       <extensions>
         <behaviorExtensions>
           <add name="BridgeAuthentication" 
                 type="Microsoft.BizTalk.Bridge.Behaviour.BridgeBehaviorElement, Microsoft.BizTalk.Bridge.Behaviour, Version=1.0.0.0, Culture=neutral, PublicKeyToken=ae58f69b69495c05" />
         </behaviorExtensions>
       </extensions>
       <behaviors>
         <endpointBehaviors>
           <behavior name="BridgeAuthenticationConfiguration">
             <!-- Enter the ACS namespace, issuer name and issuer secret of the BizTalk Services deployment -->
             <BridgeAuthentication acsnamespace="[YOUR ACS NAMESPACE]" 
                                   issuername="owner" 
                                   issuersecret="[YOUR ACS SECRET]" />
             <webHttp />
           </behavior>
         </endpointBehaviors>
       </behaviors>
       <bindings>
         <webHttpBinding>
           <binding name="BridgeBindingConfiguration">
             <security mode="Transport" />
           </binding>
         </webHttpBinding>
       </bindings>
       <client>
         <clear />
         <!--
           Go BizTalk Portal > Agreement > Send Settings > Inbound URL
           Copy the Endpoint URL and paste it in the below address field
         -->
         <endpoint name="TwoWayExternalServiceEndpointReference1" 
                   address="[YOUR EDI BRIDGE SEND URI]" 
                   behaviorConfiguration="BridgeAuthenticationConfiguration" 
                   binding="webHttpBinding" 
                   bindingConfiguration="BridgeBindingConfiguration" 
                   contract="System.ServiceModel.Routing.IRequestReplyRouter" />
       </client>
     </system.serviceModel>
   </configuration>
   
   ```
8. Обновление файла EDIBridge.config для включения настроек конфигурации
   
   * В разделе *<behaviors>* укажите пространство имен ACS и ключ, связанный с подпиской служб BizTalk.
   * В разделе *<client>* укажите конечную точку, в которой развернуто соглашение отправки EDI.
   
   Сохраните изменения и закройте файл конфигурации.
9. На панели инструментов щелкните **Соединитель** и соедините компоненты **ProcessInvoiceBridge** и **EDIBridge**. Выберите соединитель и в окне свойств задайте для параметра **Условие фильтра** значение **Match All** (Сопоставить все). Это обеспечит маршрутизацию в мост EDI всех сообщений, обрабатываемых мостом EAI.
   
   ![][10]  
10. Сохраните внесенные в решение изменения.  

### <a name="deploy-the-project"></a>Развертывание проекта
1. На компьютере, где был создан проект служб BizTalk, загрузите и установите сертификат SSL для своей подписки на службы BizTalk. В разделе служб BizTalk щелкните **Панель мониторинга**, а затем нажмите кнопку **Загрузить SSL-сертификат**. Дважды щелкните сертификат и следуйте указаниям для завершения установки. Убедитесь, что сертификат устанавливается в хранилище **Доверенные корневые центры сертификации** .
2. В обозревателе решений Visual Studio щелкните правой кнопкой мыши проект **InvoiceProcessingBridge** и выберите пункт **Развернуть**.

   ![][11]  
   
   В области вывода скопируйте конечную точку, в которой развернут мост EAI, например `https://contosowabs.biztalk.windows.net/default/ProcessInvoiceBridge`. Этот URL-адрес конечной точки понадобится вам позже.

## <a name="step-5-optional-send-edifact-invoice-in-batches"></a>Шаг 5 (необязательный). Пакетная отправка счетов EDIFACT
Мосты EDI служб BizTalk также поддерживают пакетную обработку исходящих сообщений. Эта функция полезна для принимающих партнеров, которые предпочитают получать сообщения не по отдельности, а пакетами (куда включаются сообщения, соответствующие определенным условиям).

Наиболее важным моментом пакетной обработки является фактическая отправка пакета, определяемая условиями отправки. Условия отправки могут основываться на способе получения сообщений (определяется партнером). Если включена пакетная обработка, мост EDI не отправляет исходящие сообщения принимающему партнеру до выполнения условий отправки. Например, условия отправки пакета на основе размера сообщения будут выполнены, только когда количество сообщений в пакете достигнет определенного значения. Условия также могут определяться временем отправки (например, пакет будет отправляться ежедневно в указанное время). В рамках этого решения мы зададим условия на основе размера сообщения.

1. На портале служб BizTalk щелкните соглашение, созданное ранее. Щелкните «Параметры отправки» > «Пакетная обработка» > «Добавить пакет».
2. В качестве имени пакета укажите **InvoiceBatch**, введите описание и нажмите кнопку **Далее**.
3. Укажите условия пакетной обработки, в соответствии с которыми сообщения будут включены в пакет. В этом решении будут включены все сообщения. Поэтому выберите параметр «Использовать расширенные определения» и введите **1 = 1**. Так как это условие всегда будет иметь значение Тrue, пакетной обработке будут подлежать все сообщения. Нажмите кнопку **Далее**.
   
   ![][17]  
4. Укажите условия пакетной отправки. В раскрывающемся списке выберите **MessageCountBased**, а в поле **Количество** укажите **3**. Это означает, что в компанию Northwind будет отправлен пакет, состоящий из трех сообщений. Нажмите кнопку **Далее**.
   
   ![][18]  
5. Просмотрите сводные данные и нажмите кнопку **Сохранить**. Нажмите кнопку **Развернуть** , чтобы повторно развернуть соглашение.
6. Вернитесь в приложение **Tutorial Client**, щелкните **Send In-house Invoice** (Отправить корпоративный счет) и следуйте инструкциям на экране, чтобы отправить счет. Вы увидите, что компания Northwind не получила счет, так как условие относительно размера пакета не выполнено. Повторите этот шаг еще два раза, чтобы у вас собралось три сообщения со счетами, для Northwind. Это соответствует условию отправки пакета из трех сообщений. Теперь вы увидите счет, полученный компанией Northwind.
