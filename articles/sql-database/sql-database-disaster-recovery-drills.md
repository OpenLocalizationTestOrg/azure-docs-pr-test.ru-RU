# <a name="performing-disaster-recovery-drill"></a>Отработка аварийного восстановления
Рекомендуется периодически проверять готовность приложений к рабочему процессу восстановления. Проверка поведения приложения и возможных последствий потери данных и нарушений работы, которая включает отработку отказа, предусмотрена стандартами разработки. Эта процедура также является обязательной для большинства отраслевых стандартов в рамках сертификации непрерывности бизнес-процессов.

Этапы отработки аварийного восстановления:

* моделирование сбоя уровня данных;
* восстановление;
* проверка целостности приложения после восстановления.

Рабочий процесс отработки зависит от того, как вы [спроектировали приложение для обеспечения непрерывности бизнес-процессов](sql-database-business-continuity.md). В этой статье описаны рекомендации для проведения аварийного восстановления в контексте базы данных SQL Azure.

## <a name="geo-restore"></a>Геовосстановление
Чтобы избежать возможной потери данных при проведении аварийного восстановления, выполните детализации, в среде тестирования путем создания копии в рабочей среде и его использованием для проверки рабочего процесса отработки отказа приложения.

#### <a name="outage-simulation"></a>Моделирование сбоя
Для имитации сбоя, можно переименовать базы данных-источника. Это вызовет сбой подключения приложения.

#### <a name="recovery"></a>Восстановление
* Выполните геовосстановление базы данных на другой сервер, следуя инструкциям [здесь](sql-database-disaster-recovery.md).
* Измените конфигурацию приложения, чтобы подключиться к восстановленной базе данных, и завершите восстановление, следуя инструкциям в статье [Восстановление базы данных SQL Azure или переход на базу данных-получатель при отказе](sql-database-disaster-recovery.md).

#### <a name="validation"></a>Проверка
* Проверка восстановления post целостность приложения (включая строки подключения, имена входа, основные функциональные возможности тестирования или другой проверки части процедуры signoffs стандартного приложения), чтобы завершите детализации.

## <a name="failover-groups"></a>Группы отработки отказа
Для базы данных, защищенной с помощью групп отработки отказа упражнения детализации подразумевает Плановая отработка отказа на сервере-получателе. Плановая отработка отказа гарантирует, что основной и базам данных в группу отказоустойчивого останутся синхронизации после переключения ролей. В отличие от незапланированной отработки отказа, эта операция не приводит к потере данных, поэтому отработку можно выполнить в рабочей среде.

#### <a name="outage-simulation"></a>Моделирование сбоя
Чтобы сымитировать сбой, можно отключить веб-приложение или виртуальную машину, подключенные к базе данных. Это приведет к сбоям подключения веб-клиентов.

#### <a name="recovery"></a>Восстановление
* Убедитесь, что конфигурации приложения в точек регионе аварийного восстановления на первый получателя, который становится основным полностью доступны.
* Инициировать [Плановая отработка отказа](scripts/sql-database-setup-geodr-and-failover-database-powershell.md) группы отработки отказа из сервера-получателя.
* Следуйте инструкциям в руководстве [по настройке базы данных после восстановления](sql-database-disaster-recovery.md) , чтобы завершить восстановление.

#### <a name="validation"></a>Проверка
Проверка восстановления post целостность приложения (включая подключения, основные функциональные возможности тестирования и другие проверки, необходимые для детализации signoffs), чтобы завершите детализации.
