# <a name="how-does-azure-ad-application-proxy-provide-single-sign-on"></a>Как прокси приложения Azure AD предоставляет единый вход?

Единый вход — это ключевой элемент прокси приложения Azure AD.  Он обеспечивает максимальное удобство для пользователей, так как им нужно всего лишь войти в Azure Active Directory в облаке. После аутентификации в Azure Active Directory соединитель прокси приложения осуществляет аутентификацию в локальном приложении. Внутреннее приложение не различает удаленного пользователя, входящего через прокси приложения, и обычного пользователя устройства, присоединенного к домену. 

Чтобы использовать Azure Active Directory для единого входа в приложения, необходимо выбрать **Azure Active Directory** в качестве метода предварительной аутентификации. Если выбрать **сквозной режим**, пользователи вообще не будут проходить аутентификацию Azure Active Directory, а будут направляться непосредственно к приложению. Этот параметр можно настроить при первой публикации приложения. Можно также перейти в приложение на портале Azure и изменить параметры прокси приложения. 

Чтобы просмотреть параметры единого входа, выполните следующие действия.

1. Войдите на [портале Azure](https://portal.azure.com).
2. Выберите **Azure Active Directory** > **Корпоративные приложения** > **Все приложения**.
3. Выберите приложение, параметрами единого входа которого требуется управлять.
4. Щелкните **Единый вход**.

   ![Раскрывающееся меню "Единый вход"](./media/application-proxy-sso-overview/single-sign-on-mode.png)

В раскрывающемся меню доступны пять параметров единого входа в приложение.

* Единый вход Azure AD отключен
* Единый вход по паролю
* Связанный единый вход
* Встроенная проверка подлинности Windows
* Единый входа на основе заголовка

## <a name="azure-ad-single-sign-on-disabled"></a>Единый вход Azure AD отключен

Если вы не хотите использовать интеграцию Azure Active Directory для единого входа в приложение, выберите **Единый вход Azure AD отключен**. Если этот параметр выбран, пользователи могут дважды проходить аутентификацию. Сначала они походят аутентификацию в Azure Active Directory, а затем входят в само приложение. 

Этот параметр удобно использовать, если локальному приложению не требуется аутентифицировать пользователей, но вы хотите добавить Azure Active Directory в качестве уровня безопасности для удаленного доступа. 

## <a name="password-based-sign-on"></a>Единый вход по паролю

Если вы хотите использовать Azure Active Directory в качестве хранилища паролей для локальных приложений, выберите **Вход по паролю**. Этот параметр удобно использовать, если для аутентификации приложение использует сочетание имени пользователя и пароля, а не маркеры доступа или заголовки. При использовании входа по паролю пользователям необходимо войти в приложение при первом обращении к нему. После этого Azure Active Directory предоставляет имя пользователя и пароль от имени пользователя. 

Сведения о настройке входа по паролю см. в разделе [Хранение паролей для единого входа с помощью прокси приложения](application-proxy-sso-azure-portal.md).

## <a name="linked-sign-on"></a>Связанный единый вход

Если вы уже настроили решение единого входа для своих локальных удостоверений, выберите **Вход по ссылке**. Этот параметр позволяет Azure Active Directory использовать существующие решения единого входа, но по-прежнему предоставляет пользователям удаленный доступ к приложению. 

Дополнительные сведения о функции связанного входа (существующей функции единого входа) см. в статье [Что такое доступ к приложениям и единый вход с помощью Azure Active Directory](active-directory-appssoaccess-whatis.md#how-does-single-sign-on-with-azure-active-directory-work).

## <a name="integrated-windows-authentication"></a>Встроенная проверка подлинности Windows

Если ваши локальные приложения используют встроенную проверку подлинности Windows (IWA) или вы хотите использовать единый вход на основе ограниченного делегирования Kerberos (KCD), выберите **Встроенная проверка подлинности Windows**. Если выбран этот параметр, то пользователям достаточно пройти аутентификацию в Azure Active Directory, после чего соединитель прокси приложения олицетворяет пользователя, чтобы получить маркер Kerberos и выполнить вход в приложение. 

Сведения о настройке встроенной проверки подлинности Windows см. в разделе [Реализация единого входа в приложения с помощью прокси приложения](active-directory-application-proxy-sso-using-kcd.md).

## <a name="header-based-sign-on"></a>Единый входа на основе заголовка 

Если приложения используют заголовки для аутентификации, выберите **Вход на основе заголовков**. Если выбран этот параметр, пользователям необходимо только пройти аутентификацию Azure Active Directory. Корпорация Майкрософт сотрудничает с разработчиками сторонней службы аутентификации PingAccess, которая преобразовывает маркер доступа Azure Active Directory в формат заголовка для приложения. 

Сведения о настройке аутентификации на основе заголовка см. в разделе [Публикация приложений с поддержкой аутентификации на основе заголовков с использованием прокси приложения Azure AD и PingAccess](application-proxy-ping-access.md).
